# Раздел доклада: сильные паттерны работы с AI-agent

> Переосмысленная версия блока, ранее называвшегося «29–53 мин. Сильные паттерны: 8 экспертных приёмов». Заменяет текущее содержание блока в `[[AI-Agent Delivery Workflow]]`. Готовь к чтению вместе с `[[Speaker Notes]]` и `[[Live Demo Script]]`.

## 1. Диагностика старого раздела

Старый блок звучал как мастер-класс на бумаге, но при близком чтении проседал по нескольким измерениям. Здесь — что именно ломалось, с конкретикой по каждому пункту.

### 1.1. Приукрашенные и непроверенные цитаты

Сделана независимая верификация каждой цитаты в старом разделе. Результат:

| Утверждение | Реальность |
|---|---|
| Anthropic «Building Effective Agents» **прямо называет** этот шаблон `gate` (про approval-checkpoint) | Слово «gate» в статье есть, но это **программная проверка между LLM-вызовами**, не human-checkpoint. Термин «approval gate» в статье **отсутствует**. |
| `single highest-leverage thing you can do` (Verification Oracle) | **Не найдено** ни в одной публикации Anthropic. |
| Karpathy на Sequoia 2026: «context window — RAM, файлы — disk» | В опубликованном Karpathy саммари [Sequoia Ascent 2026](https://karpathy.bearblog.dev/sequoia-ascent-2026/) этой формулы **нет**. Реальный тезис Karpathy с Sequoia 2026 — про verifiability, не про память. |
| Simon Willison Red/Green TDD: тесты — это **оракул** | Глава существует, но слово «oracle» в ней не используется. |
| Eugene Yan: structured output снижает error rate на **десятки процентов** | Цифры в статье нет. Качественный тезис — да. |
| Devin / Cognition — **два mandatory checkpoint** | Не подтверждено публичными материалами Cognition. |
| EU AI Act Article 12, enforcement **август 2026** | Дата верна, но 19 ноября 2025 EC опубликовал Digital Omnibus с предложением переноса high-risk обязательств на 2 декабря 2027. Без этой оговорки — устаревшая информация. |

**Вывод:** доклад заявлял инженерный мастер-класс, но опирался на цитаты, которые при первом гуглении разваливались. На enterprise-аудитории такой блок выглядит как fact-stretching, а не экспертиза.

### 1.2. Структурные проблемы

**A. Имитация авторитетов вместо разбора паттернов.** Восемь паттернов следовали одному шаблону: «definition / bad pattern / real example / **цитата авторитета** / recovery». Зал слышал ссылки на Anthropic + Karpathy + Willison + Yan **восемь раз подряд**. Это перегруз и читалось как защита диссертации, а не мастер-класс.

**B. Слабая связь с opencode.** opencode упоминался 4–5 раз, но уникальные особенности инструмента не использовались:
- `Plugin-based hooks` (~25 lifecycle events) — не подавались как отдельный паттерн.
- `Skills` как primitive — упоминались в блоке 4–9, но в Patterns ни слова.
- Встроенные `Build / Plan / Explore / Scout` агенты — не использовались в Pattern про subagents.
- Приоритет `AGENTS.md > CLAUDE.md` — пропущен.

**C. Демо-анкеры слабы.** Старый раздел декларировал «3 паттерна вы видели в demo» (Context, Plan-Gate, Verification). Но:
- Pattern «Context engineering» в demo **не показан** — нет `/clear` или externalization.
- Pattern «Output structure» — в demo есть MR description, но не подан как machine-readable schema.
- Pattern «Attribution» — в demo появляется мельком в шаге 12.

**D. Pattern «Verifiability boundary» в финальной позиции — слабая идея.** Это философский метатезис. На enterprise-аудитории он сядет хорошо, но через философию, а не через инженерию. Финальный — худшая позиция, потому что после него идут Risks, и philosophical mood разбивается.

**E. Pattern «Attribution в MR» среди инженерных паттернов — неуместно.** Это GRC-тезис, его место в Risks, не среди приёмов кодовой работы.

**F. «Real Stories» — generic-сценарии под реальные кейсы.** Story 1 («рефакторил обработчик ошибок в `[SERVICE_A]`»), Story 2 («retry в `[MODULE_B]`»), Story 4 («bug fix в `[LEGACY_MODULE]` через рефлексию»), Story 8 («authorization model для multi-tenant») — это **типовые learning examples**, не конкретные кейсы. Зал, который сам ходил в этот код, распознал бы за 30 секунд. Самая опасная часть старой версии.

### 1.3. Что сохраняется в новой версии

По сути верны:
- ✅ **Plan-Gate-Execute** — концепция верна, opencode явно поддерживает (Plan vs Build), demo показывает в шаге 8.
- ✅ **Verification Oracle** — концепция верна, opencode hooks для этого, demo шаги 10–11.
- ✅ **Subagent / Writer-Reviewer** — отличный паттерн, у opencode primitive.
- ✅ **AGENTS.md** — реальный, opencode поддерживает, GitHub в 2025 опубликовал анализ 2500 репозиториев.
- ✅ **Context engineering** — концепция верна, источник чистый.

Уходят из паттернов:
- ❌ **Attribution** → переезжает в Risks.
- ❌ **Verifiability boundary** → становится финальной мыслью раздела, не отдельным паттерном.

Заменяются и переименовываются:
- **Output structure** — без выдуманных цифр Eugene Yan; вместо них — Anthropic про guardrails как первую линию контроля.
- **Hooks as deterministic gates** — новый паттерн на Level 4, использует уникальную особенность opencode (plugin-based hooks).

## 2. Новая логика раздела

### 2.1. Каркас — Maturity ladder

Раздел построен как лестница из 4 уровней зрелости работы с AI-агентом. Каждый уровень: что характеризует пользователя, какие боли он уже решил, что нерешено. Слушатель в любой момент может ответить себе на вопрос: «я где?».

| Уровень | Описание | Что появляется на этом уровне |
|---|---|---|
| **Level 1 — Промптовый пользователь** | Стартовая точка. Один промпт — один результат. Через неё проходят все. | — (это **фрейм**, не паттерны) |
| **Level 2 — Контекстный пользователь** | Управляет тем, что агент видит. RAM, project memory. | Pattern A — Context engineering. Pattern B — AGENTS.md. |
| **Level 3 — Workflow-пользователь** | Структурирует процесс: phase split, verification, isolation. | Pattern C — Plan mode. Pattern D — Verification Oracle. Pattern E — Subagent + Writer/Reviewer. |
| **Level 4 — Team playbook owner** | Закрепляет workflow в коде команды. От личной дисциплины — к командному стандарту. | Pattern F — Hooks as deterministic gates. Pattern G — Output structure as contract. |

**Итого:** 7 паттернов (по 2-3 на уровень после Level 1).

### 2.2. Принципы подачи

1. **Без оценочных суждений.** Level 1 — не «вы это делаете неправильно». Это «через эту точку проходят все, и это нормально». Доклад не критикует Kilo Code и не указывает залу, что они делают плохо.
2. **Каждый паттерн закрывает конкретную узнаваемую боль команды.** Без боли — паттерн в воздухе. Боли подобраны из тех, что слушатели подтверждают.
3. **Источники — только проверенные.** Никаких приукрашенных цитат и выдуманных чисел. Если внешний тезис нужен — даётся буквально, со ссылкой; если такого тезиса в источнике нет — не используется вообще.
4. **opencode-native primitives используются явно.** AGENTS.md, Plan mode, hooks, subagents — каждый паттерн опирается на реальные возможности инструмента, демонстрируемого в зале.
5. **Связь с demo — прямая.** Каждый паттерн уровней 2–4 имеет либо demo callback, либо живой фрагмент (показ AGENTS.md, MR template).

### 2.3. Финальная мысль раздела — verifiability boundary

Verifiability boundary не подаётся как отдельный паттерн — это рамка, через которую слушатель оценивает любую будущую задачу. Озвучивается **один раз, в outro раздела**:

> Перед каждой задачей — один вопрос: «как я узнаю, что результат правильный?». Если есть тест, схема, линтер, build, который ответит — агенту можно дать автономию. Если нет — агент не ускоряет, он маскирует риск. Знать, где эта граница — это и есть зрелая дисциплина работы с агентом.

Это даёт залу ментальный инструмент, который останется с ними после доклада.

### 2.4. Take-away для зала

Одной фразой:

> **«Сильный пользователь — это не промпты. Это workflow и playbook. Maturity ladder показывает путь — Pilot 4 недели запускает первое движение по нему».**

Это естественная подводка к CTA.

## 4. Полный текст для выступления

Полный текст ниже — то, что произносится со сцены. Оптимизировано под чтение вслух (короткие фразы, паузы маркированы тире). Реальные имена сервисов / модулей / endpoint'ов спикер подставляет перед выступлением (помечены `[ВПИСАТЬ: ...]`).

### Intro + Level 1 (1:30 мин)

Дальше — главный экспертный блок раздела. Я не буду давать вам список из десяти паттернов в стиле «топ советов по AI». Вместо этого — лестница из четырёх уровней зрелости. Где находится сильный пользователь, что он делает, и как ваша команда туда дойдёт.

Через четыре минуты вы сможете честно ответить себе на вопрос: «я на каком уровне». А через двадцать четыре — у вас будет карта, как двигаться выше.

**Level 1. Промптовый пользователь.** Это базовая точка. Через неё проходят все. Я через неё проходил. Скорее всего, и вы. Здесь работа с агентом выглядит просто: тикет → один промпт → один большой результат. Нет уровневой постановки. Нет фаз. Нет проверок до коммита. Просто «дай задачу — получи код».

Это не «плохо». Это **точка отсчёта**. Сильные команды отсюда стартуют. Вопрос — куда они движутся дальше.

Симптомы Level 1, которые вы могли видеть на своём проекте: двести-строчные PR от агента вместо тридцати; свободный текст в MR description, который ревьюер реверс-инжинирит из diff; повторение одних и тех же конвенций в каждом промпте. Если узнаёте — это просто значит, что у вас впереди уровни 2, 3, 4. Сейчас покажу, что на каждом из них появляется.

### Level 2.A — Context engineering (3 мин)

**Что появляется на этом уровне.** Управление тем, что агент видит, как ресурсом. Не «лучший промпт» — а контролируемый контекст.

**Какую боль закрывает.** Узнаваемая ситуация: вы сорок минут в сессии, агент написал работающий код — и в финале возвращает результат, который **противоречит вашему первому требованию**. Не «забыл», не «глупый» — контекст забился до того, что раннее требование вытеснилось.

В demo это не показано напрямую — там сессия слишком короткая. Но в вашей реальной работе это случается каждый раз, когда сессия идёт дольше двадцати минут. Anthropic в статье «Effective Context Engineering for AI Agents» (сентябрь 2025) формулирует это буквально: задача — найти **минимальный набор high-signal токенов**, который максимизирует вероятность нужного результата.

**Как это выглядит на практике.**

- Сессия больше 20 минут или больше трёх итераций — `/clear`. Перезагрузить только то, что нужно сейчас.
- Длинные failed attempts, stack traces, экспериментальные diff'ы — выносить в файлы, не держать в активном контексте.
- Перед началом новой подзадачи — отдельная сессия, не продолжение предыдущей.

**Demo callback — Шаг 6.** Помните, в demo MCP к OpenAPI вернул impact-карту за полторы минуты. Это сработало не потому, что агент быстрый. Это сработало потому, что MCP **передал в контекст структурированный контракт**, а не сорок страниц YAML. Структурированный контекст — это и есть минимальный набор high-signal токенов про API. Это context engineering в действии.

**Опора в opencode.** Plan-mode и Build-mode — два разных профиля контекста. Plan читает много, но edits заблокированы — контекст не разрастается diff'ами. Build переключается, когда уже знаем, что менять. Эта механика поддерживает context engineering на уровне инструмента.

**Take-away паттерна.** Контекст — это RAM. Профилируйте до того, как заметите, что он переполнен.

### Level 2.B — AGENTS.md как project memory (3 мин)

**Что появляется на этом уровне.** Конвенции вашего проекта переезжают **из головы — в файл**. Файл коммитится в git и становится частью кодовой базы. Агент читает его в начале каждой сессии.

**Какую боль закрывает.** Узнаваемая боль: каждый разработчик в команде работает с агентом по-своему. Один пишет в каждом промпте «у нас Spring Boot, Java 17, Lombok запрещён». Второй не пишет. Через два месяца стиль кода в репо начинает расходиться — где-то Lombok, где-то нет; разные паттерны логирования; разный naming для DTO. Ревью на каждом MR — комментарий «у нас так не делают».

**Как это выглядит на практике.**

В opencode файл называется `AGENTS.md`. Он лежит в корне репо, **коммитится в git**. Всё, что туда кладётся, — это инструкции, **которые агент сам не выведет из кода**:

- build-команды (как собирать, как запускать тесты);
- code style (запрещённые библиотеки, naming для DTO/Service/Controller);
- security-ограничения (что нельзя трогать без явного разрешения);
- конвенции логирования и метрик;
- ссылки на ADR-ы критичной архитектуры.

GitHub Engineering в 2025 опубликовал [анализ 2500 репозиториев](https://github.blog/ai-and-ml/github-copilot/how-to-write-a-great-agents-md-lessons-from-over-2500-repositories/) с лучшими практиками AGENTS.md. Главный вывод: успешные файлы — короткие и специфичные. *«Один реальный сниппет вашего стиля бьёт три параграфа описания».*

**[ЖИВОЙ ФРАГМЕНТ — 30 СЕКУНД]** Вот, например, мой текущий AGENTS.md для проекта `[ВПИСАТЬ: имя репо]`. *(показ фрагмента на экране 30 секунд — спикер открывает файл из своего рабочего репозитория и проходит по 3-4 ключевым секциям)*. Это стартовая версия — личная, у меня в working tree. Командная версия — следующий шаг: согласовать содержимое с коллегами и закоммитить. Это **то, с чего я предложу стартовать pilot'у**.

**Опора в opencode.** opencode официально поддерживает `AGENTS.md` — приоритет `AGENTS.md > CLAUDE.md`, генерация через `/init`, scope: project root + `~/.config/opencode/AGENTS.md` для глобальных правил.

**Take-away паттерна.** Конвенции — на диске, в git, как часть кодовой базы. Без этого AI-workflow в команде расходится через два месяца.

### Level 3.A — Plan mode: Explore → Plan → Implement (3:30 мин)

**Что появляется на этом уровне.** Разделение фаз. Сначала агент **только читает и предлагает план** — никаких правок файлов. Потом — approval. Только после approval — переключение в режим, где агент пишет код.

**Какую боль закрывает.** Узнаваемая боль: агент возвращает PR на двести строк, когда задача требовала тридцать. Половина — generic-абстракции, которые в проекте не приживутся. Ревьюер тратит час на «понять, что это за рефакторинг», вместо двух минут на ревью локального изменения.

**Demo callback — Шаг 8.** В demo вы видели момент: я попросил план реализации, **не код**. Агент вернул пять-семь пунктов с файлами и порядком. Я сказал: «шаги 1–3, тесты отдельно». Получил тридцать строк, не двести. Это и есть Plan mode в действии. План — самый важный шаг, на который большинство забивают.

**Как это выглядит на практике.** Anthropic в [Claude Code best practices](https://code.claude.com/docs/en/best-practices) формулирует напрямую: *«Letting Claude jump straight to coding can produce code that solves the wrong problem. Use plan mode to separate exploration from execution».* Четыре фазы: Explore (read-only) → Plan → Implement → Commit. Между Plan и Implement — **ваш approval**.

**Опора в opencode.** В opencode Plan и Build — это отдельные встроенные primary-агенты с разными permission-профилями. Plan — file edits и bash установлены в `ask` по умолчанию. Build — все инструменты `allow`. Переключение — Tab в TUI. **Это не настройка, это архитектура инструмента**.

**Что ещё важно.** План становится письменным артефактом. Он остаётся в сессии, на него можно сослаться, его можно показать коллеге. Когда вы читаете план до кода — вы не платите за чужое решение. Это и есть архитектура работы с агентом.

**Take-away паттерна.** План — отдельная фаза. Approval — отдельная фаза. Код — отдельная фаза. Граница между ними — это и есть архитектура.

### Level 3.B — Verification Oracle (3 мин)

**Что появляется на этом уровне.** Агент **сам запускает** тесты, build, линтер, OpenAPI-валидатор — и читает результат. Не «агент написал — я запустил — агент починил» с человеком как ботом-передатчиком. Замкнутая петля.

**Какую боль закрывает.** Случай, который случается у всех: агент возвращает «правдоподобный код». Компилируется, выглядит корректно. На интеграционном тесте — падает. Узнаёте через CI на следующий день. Час на дебаг + два часа на переделку.

**Demo callback — Шаги 10 и 11.** В demo агент написал тесты, прошёл self-review своего диффа и нашёл NPE-риск, который сам же создал две минуты назад. Это не магия. Это **verification loop с тестами как оракулом**. Не верит объяснению — запускает.

**Как это выглядит на практике.** Anthropic в Claude Code best practices: *«Claude performs dramatically better when it can verify its own work, like run tests, compare screenshots, and validate outputs. Without clear success criteria, it might produce something that looks right but actually doesn't work».*

Никогда не давайте агенту задачу без verifiable success criterion. Если ответ на вопрос «как мы узнаем, что результат правильный?» — «никак» или «протестируем вручную потом», — вы не делегируете агенту, вы маскируете риск. Defining characteristic of coding agent — не то, что он генерирует код. А то, что он может его запустить и прочитать результат.

**Опора в opencode.** Через MCP агент имеет программный доступ к bash — может выполнять `mvn test`, `gradle build`, `npm test`. Через permissions: `"mvn test": "allow"`, `"git push *": "deny"` — verification-команды разрешены, деструктивные — запрещены.

**Take-away паттерна.** Тест, схема, линтер, build — это и есть оракул, который говорит агенту, когда он закончил. Без оракула — у вас не agent, у вас chat.

### Level 3.C — Subagent + Writer/Reviewer (3 мин) [cuttable]

**Что появляется на этом уровне.** Изолированные сессии под отдельные задачи: investigation, implementation, review — каждая со своим чистым контекстом. Тот же агент, что писал код, его проверять **не должен** — он предвзятый ревьюер по определению.

**Какую боль закрывает.** Узнаваемое: одна гигантская сессия, в которой агент исследовал, реализовал, и сам себя проверил. Главный context забит investigation, failed attempts, экспериментальными правками. Self-review в этом контексте — формальность.

**Demo callback — Шаг 3.** Помните, в demo на шаге Questions to business агент вернул вопросы, среди которых был тот, что я мог пропустить? Это работало именно потому, что агент посмотрел на тикет **с чистого контекста как pair-аналитик**. Не как тот, кто уже три минуты копался в коде. Тот же эффект — на review.

**Как это выглядит на практике.** Anthropic в Claude Code best practices ([URL](https://code.claude.com/docs/en/best-practices)) описывает Writer/Reviewer pattern буквально:

| Session A (Writer) | Session B (Reviewer) |
|---|---|
| `Implement a rate limiter for our API endpoints` | (отдельная сессия с fresh context) |
| | `Review the rate limiter implementation. Look for edge cases, race conditions, consistency with our existing middleware patterns.` |

Reviewer работает в свежем контексте — нет bias к коду, который он не писал. Это принципиально отличается от self-review той же сессии.

**Опора в opencode.** opencode из коробки умеет subagents через Task tool. Встроенные агенты — Build, Plan, General, Explore, Scout. Можно создавать свои в `.opencode/agents/<name>.md`. Subagent работает в **изолированной сессии с собственным toolset и permission-профилем**. Конкурентный запуск возможен — несколько параллельных subagents под разные исследования.

**Take-away паттерна.** Subagent — это не параллелизм ради скорости. Это **изоляция контекста ради качества**.

### Level 4.A — Hooks as deterministic gates (2:30 мин) [cuttable]

**Что появляется на этом уровне.** Команда переходит от **рекомендаций** к **гарантиям**. Hooks — это правила, которые срабатывают **на каждом событии**, без исключений.

**Какую боль закрывает.** Узнаваемое: правила в AGENTS.md есть, но соблюдаются по настроению. Кто-то торопится — пропускает прогон тестов перед коммитом. Кто-то забывает запустить линтер. Один MR на ревью — нет attribution к AI. Полгода спустя — внутренний security review, и реконструировать, какой код agent-assisted, никто не может.

**Как это выглядит на практике.** Anthropic в Claude Code best practices формулирует напрямую: *«Unlike CLAUDE.md instructions which are advisory, hooks are deterministic and guarantee the action happens».*

Конкретные примеры hooks для enterprise Java-проекта:

- `tool.execute.before` на любой `git commit` → автоматически прогнать `mvn test`. Если красные — заблокировать коммит.
- `file.edited` на `*.sql` миграции → автоматически валидировать через линтер.
- `tool.execute.after` на любой edit → пересчитать coverage, обновить локальный отчёт.
- `permission.asked` на действия в auth-модуле → требовать дополнительный approval.

**Опора в opencode.** opencode реализует hooks через **plugin system**: JS/TS файлы в `.opencode/plugins/`. Около 25 lifecycle events: `tool.execute.before`, `tool.execute.after`, `session.created`, `session.idle`, `session.compacted`, `file.edited`, `permission.asked` и другие. Pluggable, версионируется в git, выполняется на машине разработчика, не на сервере opencode. Это даёт enterprise-grade контроль без зависимости от vendor lock-in.

**Take-away паттерна.** Конвенции — рекомендация. Hooks — закон. На уровне команды — закон масштабируется, рекомендация теряется.

---

### Level 4.B — Output structure as contract (3 мин)

**Что появляется на этом уровне.** Output агента — не свободный текст, а **interface contract** с downstream pipeline. Не «напиши summary», а «верни JSON по этой схеме» или «верни Markdown с этими разделами».

**Какую боль закрывает.** Подтверждённая боль: MR description'ы — свободный текст, у каждого свой формат. Ревьюер открывает PR и сразу попадает в реверс-инжиниринг: что изменено, зачем, что проверили, какие риски. Время ревью — больше, чем должно быть. Никакая автоматизация поверх MR description **невозможна** — нечего парсить.

**Demo callback — Шаг 12.** В demo агент сгенерировал MR description по строгому шаблону: что изменено / затронутые файлы / тесты / риски / AI-attribution. Это не decoration — это **machine-readable artifact**. Помимо ускорения ревью, на нём можно построить:

- CI hook, который парсит секцию «Риски» и блокирует merge при `risk_level: high` без approval от tech lead.
- Дашборд, который собирает по всем MR процент кода, отмеченного AI-assisted.
- Автоматический tagging для security review.

**Как это выглядит на практике.**

Возьмите три типовых задачи, которые в команде повторяются (impact analysis, MR summary, code review feedback) — и **зафиксируйте output** как структуру. В opencode это становится `command`:

```
.opencode/commands/mr-summary.md
.opencode/commands/impact.md
.opencode/commands/review.md
```

Каждая команда — параметризованный шаблон промпта. Один формат. Один и тот же output во всей команде.

Anthropic называет такие structured output [«guardrails as a first line of control»](https://www.anthropic.com/engineering/building-effective-agents). Eugene Yan в [LLM Patterns](https://eugeneyan.com/writing/llm-patterns/) — *«Apply guidance whenever possible. It provides direct control over outputs and offers a more precise method to ensure that output conforms to a specific structure»*. Без чисел, но качественно — точно.

**Опора в opencode.** Commands в `.opencode/commands/` — это и есть механизм фиксации structured output. Параметры через `$ARGUMENTS`, `$1/$2/$3`. Файл-инъекция через `@path`. Команды живут в git, у всех в команде одинаковые.

**Take-away паттерна.** Output — это interface. Структура output — это контракт между агентом и delivery pipeline.

---

### Outro: verifiability boundary + take-away (1:30 мин)

Семь паттернов. Четыре уровня. Если бы я свёл всё это в одну ментальную рамку — она была бы такой.

Karpathy на Sequoia Ascent 2026 сформулировал ([URL](https://karpathy.bearblog.dev/sequoia-ascent-2026/)): *«Traditional software automates what you can specify. LLMs and reinforcement learning automate what you can verify».*

Перевожу на ваш delivery pipeline. Перед каждой задачей — задавайте себе один вопрос:

> Как я узнаю, что результат правильный?

- Если есть тест, схема, линтер, build, который ответит на этот вопрос — агент закроет свой loop сам. Это high-verifiability задача. Здесь работает Plan mode + Verification Oracle, и вы экономите часы.
- Если ответа нет — это low-verifiability задача. Архитектурные решения, проектирование security model, дизайн интеграций. Здесь агент **не ускоряет процесс** — он **маскирует риск**. Тут нужен человек на каждом output.

Знать, где эта граница, — это и есть зрелая дисциплина работы с агентом. Не «как чаще использовать AI», а «где границу провести».

И финальная мысль раздела.

> **Сильный пользователь — это не промпты. Это workflow и playbook. Maturity ladder показывает путь. Pilot 4 недели запускает первое движение по нему.**

Дальше — риски и контроль качества.

## 10. Итоговый checklist для команды

> Раздаточный материал. Печатается на одной странице. Команда забирает с собой.

### Самооценка по Maturity ladder

**Level 1 — Промптовый пользователь.** Через это проходят все.

- [ ] Большинство задач — один промпт, один большой результат
- [ ] PR от агента часто > 100 строк
- [ ] MR description — свободный текст, у каждого свой формат
- [ ] Конвенции повторяются в каждом промпте

**Level 2 — Контекстный пользователь.** Контролируете контекст.

- [ ] Длинные сессии (>20 мин) ловите по `/clear` или вынесением в файл
- [ ] Есть `AGENTS.md` (личный или командный) с build-командами, code style, security
- [ ] AGENTS.md закоммичен в git, читается агентом каждую сессию
- [ ] Стиль кода после AI-assisted изменений в репо не расходится между разработчиками

**Level 3 — Workflow-пользователь.** Структурируете процесс.

- [ ] Используете Plan mode (Explore → Plan → Implement) — план до кода
- [ ] Approval плана — фиксированная фаза, не пропускается
- [ ] Каждый AI-assisted change имеет verifiable success criterion (тест, схема, build)
- [ ] Агент сам запускает тесты/build/линтер и читает результат
- [ ] Используете отдельные subagents для investigation / реализации / review
- [ ] Reviewer работает в fresh context, не в той же сессии, что писала код

**Level 4 — Team playbook owner.** Закрепляете workflow в коде.

- [ ] Есть hooks в `.opencode/plugins/`, гарантирующие тесты на каждом коммите
- [ ] Деструктивные операции запрещены через permissions, не через дисциплину
- [ ] MR description — структурированный шаблон, единый для всей команды
- [ ] Шаблон зафиксирован как `command` в opencode (`/mr-summary`)
- [ ] CI парсит секции структурированного output (риски, attribution, тесты)
- [ ] Регулярно обновляете AGENTS.md по результатам анализа прошлых сессий

### Mental tool на каждую задачу

> Перед тем как делегировать задачу агенту — спросите себя:
>
> **Как я узнаю, что результат правильный?**
>
> Есть тест/схема/build, который ответит → агент закроет свой loop сам.
>
> Ответа нет → агент не ускоряет процесс, он маскирует риск. Нужен человек на каждом output.

### Что делать дальше

- Если вы на Level 1 — вступайте в pilot. 4 недели, и вы будете на Level 2-3.
- Если вы уже на Level 2 — запускайте AGENTS.md в команде на этой неделе. 2 часа работы, неделю на консолидацию.
- Если вы на Level 3 — разберитесь с hooks для вашего стека на следующем спринте. Это инвестиция на полгода вперёд.
- Если вы на Level 4 — готов послушать ваш опыт после доклада.
