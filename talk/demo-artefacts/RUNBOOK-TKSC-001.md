# Workflow аналитика — TKSC-001

Прагматичный пошаговый flow: от полученной бизнес-задачи до готового handoff'а для разработки. Каждый шаг — **Вход / Промпт / Выход**. Контекст накапливается от шага к шагу.

---

## Стартовый контекст

Что у аналитика на руках, когда он садится за работу:

### 1. Текст задачи

```
TKSC-001 — Унифицировать показатель «осталось дней» grace-периода

Бизнес-проблема:
Потребители получают разные значения «осталось дней» до окончания
grace-периода — расхождение до одного дня. Поддержка получает
регулярные обращения, доверие к показателю падает.

Бизнес-требование:
Источником истины для значения «осталось дней» должен стать сервер.
Потребители только отображают полученное значение, самостоятельный
пересчёт убираем.

Acceptance:
- У всех потребителей одинаковое значение «осталось дней».
- Поведение для уже истёкшего grace согласовано с продуктом.
- Существующие потребители продолжают работать без обновления
  (backwards compatibility ответа API).
```

### 2. Сервис и точка интеграции

- **Сервис:** `packagesearch` (внутренний движок подбора пакетов услуг и тарифов).
- **Endpoint:** `POST /FL/gracePeriod` — возвращает информацию о grace-периоде клиента.
- **Текущий контракт response (на бизнес-уровне):** даты начала/конца grace, длительность (код типа «14D»). Поля «осталось дней» **в response нет** — потребители вычисляют локально, отсюда расхождение.

## Шаг 1 — Вопросы к продукту

### Вход

- Текст задачи TKSC-001.
- Сервис `packagesearch`, endpoint `POST /FL/gracePeriod`.
- Знание: в текущем response поля «осталось дней» нет.

### Промпт

```text
Я аналитик. Работаю над задачей  @tickets\TKSC-001.md на сервисе packagesearch.

Задача — унифицировать показатель «осталось дней»: сервер начинает
возвращать это значение, потребители только отображают. Текущий
response endpoint'а включает даты начала/конца grace и длительность,
но НЕ включает «осталось дней» — потребители вычисляют локально,
отсюда расхождение до одного дня.

Сгенерируй 4-6 вопросов к Product Owner до начала работы над ФС.
Требования:
- Каждый вопрос — про конкретный пользовательский сценарий, не про
  реализацию.
- 2-4 варианта ответа на каждый (A/B/C/D) в бизнес-формулировке,
  с короткой иллюстрацией «тогда в ситуации X значение будет Y».
- Один вопрос помечен ★ как «missed question» — тот, который
  НЕ покрыт исходным текстом задачи и угадать ответ опаснее всего.
- В конце — короткая секция «Ожидаемый формат ответа от продукта».

Сохрани в tickets/TKSC-001-questions.md. Не предлагай схему API.
Не упоминай имена полей или DTO.
```

### Выход

`tickets/TKSC-001-questions.md`:
- 4-6 вопросов через `> [!todo]` callouts.
- У каждого — 2-4 варианта ответа A/B/C/D с иллюстрациями.
- Один помечен ★ (для grace-периода обычно — сценарий «истёкший grace»).
- Секция «Ожидаемый формат ответа от продукта».

После встречи с PO — добавить под каждый вопрос парный `> [!info] <Имя>, <дата>` с выбранным вариантом и обоснованием в 1 строку.

## Шаг 2 — Refined AC + edge cases

### Вход

- `tickets/TKSC-001-questions.md` с ответами продукта.
- Стартовый контекст (сервис, endpoint, текущий контракт response).

### Промпт

```text
На основе ответов продукта в tickets/TKSC-001-questions.md (> [!info]
callouts) и задачи tickets/TKSC-001.md создай tickets/TKSC-001-refined-fs.md.

Контекст: задача TKSC-001 — добавить в response endpoint'а
POST /FL/gracePeriod сервиса packagesearch значение «осталось дней».
Backwards compatibility — критично (существующие потребители
не должны ломаться).

Разделы:

1. Refined Acceptance Criteria — нумерованный список 7-9 пунктов.
   Каждый AC — observable behavior, не реализация. БЕЗ имён полей,
   типов, endpoint'ов в тексте AC. Каждый AC должен учесть минимум
   один ответ продукта из questions.md.

2. Edge cases (для test-плана) — таблица:
   | # | Сценарий | Ожидаемое значение | Что видит потребитель |
   Минимум 6 строк. Обязательно — граничные случаи полуночи UTC
   (запрос за минуту до и через минуту после).

3. Out of scope — что НЕ делаем в этой задаче.

4. Open questions (non-blocking) — то, что не блокирует, но
   требует follow-up.

5. Что разработчику решать самому — имя поля в response,
   технический тип (Integer / nullable / Optional / sentinel),
   расположение логики вычисления, способ сериализации
   «отсутствия значения». Минимум 5 пунктов.
```

### Выход

`tickets/TKSC-001-refined-fs.md`:
- 7-9 AC в бизнес-формулировке.
- Таблица edge cases ≥ 6 строк, включая UTC boundary.
- Разделы: Out of scope, Open questions (non-blocking), Что dev решает сам.
- Никаких имён полей, типов, endpoint'ов в AC.

## Шаг 3 — Архитектура взаимодействия (C4 Container)

### Вход

- `tickets/TKSC-001-refined-fs.md`.
- Стартовый контекст (сервис, endpoint).

### Промпт

```text
Ты сильный solution architect и эксперт в построение диаграмм в нотации c4 уровня контейнера 
Создай @tickets/TKSC-001-architecture.md.

Контекст: задача TKSC-001 — добавление «осталось дней» в response
endpoint'а POST /FL/gracePeriod сервиса packagesearch.
Потребители читают значение и используют его (не вычисляют).
Файлы: @tickets/TKSC-001.md - задача от бизнеса
@tickets/TKSC-001-refined-fs.md
@tickets/TKSC-001-questions.md

Содержание:

1. Контекст — 1-2 фразы: что показывает диаграмма.

2. C4 Container диаграмма — Mermaid в нотации C4Container.

Участники:
- Потребитель A (downstream-сервис / канал) — читает значение
  «осталось дней» из response и использует
- Потребитель B (второй downstream-канал) — то же
- Backend packagesearch — рассчитывает значение и возвращает
  в response endpoint'а /FL/gracePeriod
- Справочник конфигураций — хранит правила льготного периода
- Биллинг-сервис (External) — источник дат начала/конца grace,
  если применимо

Связи (Rel):
- Потребитель A → packagesearch: POST /FL/gracePeriod (HTTPS)
- Потребитель B → packagesearch: POST /FL/gracePeriod (HTTPS)
- packagesearch → Справочник: читает правила
- packagesearch → Биллинг: получает даты границ (если есть)

3. Границы ответственности — таблица:
   | Container | За что отвечает | За что НЕ отвечает |
   Минимум 4 строки.

4. Связь с refined AC — короткая таблица:
   | Container | Закрывает AC |

ВАЖНО:
- Бизнес-уровень. Без имён классов, методов, DTO, схем таблиц.
- Технологии в скобках — только если часть бизнес-контракта
  (HTTPS — да; Spring Boot 3 / PostgreSQL — нет).
- Endpoint /FL/gracePeriod указывается в подписи связи (это
  публичный контракт, не code ref).

```

### Выход

`tickets/TKSC-001-architecture.md`:
- Mermaid `C4Container` блок, рендерится в Obsidian.
- 4-5 Container'ов, минимум 4 связи Rel.
- Endpoint `/FL/gracePeriod` упомянут на связях (как часть бизнес-контракта).
- Таблица «Границы ответственности» ≥ 4 строки.
- Таблица «Связь с refined AC».

## Шаг 4 — Handoff пакет для разработки

### Вход

- `tickets/TKSC-001-questions.md` (с ответами).
- `tickets/TKSC-001-refined-fs.md`.
- `tickets/TKSC-001-architecture.md`.
- Стартовый контекст.

### Промпт

```text
Собери финальный handoff в FS/TKSC-001-handoff.md.

Frontmatter:
type: handoff
ticket_id: TKSC-001
status: draft
analyst: <Имя>
handoff_date: <сегодня>

Разделы:

1. Бизнес-контекст — 1-2 фразы.

2. Что меняется — кратко: добавляется значение «осталось дней» в
   response endpoint'а POST /FL/gracePeriod сервиса packagesearch.
   Запрос не меняется. Backwards-compatible.

3. Refined AC — скопировать из refined-fs.md без изменений.

4. Edge cases — буллет-лист (полная таблица — в refined-fs.md).

5. Architecture (C4 Container) — embed Mermaid диаграмму из
   tickets/TKSC-001-architecture.md.

6. Открытые вопросы:
   6.1 Закрытые — > [!info] callouts с автором и датой.
   6.2 Non-blocking — > [!todo].
   6.3 Блокирующие — должно быть ПУСТО.

7. Что разработчику решать самому — 5+ пунктов (имя поля, тип,
   сериализация null, расположение логики, внутренний рефакторинг
   если scope не превышен).

8. Signoff — таблица: Аналитик | Имя | Дата;
   Продукт | Имя | Дата;
   Архитектор | Имя или прочерк | Дата.

В конце — > [!success] Handoff is ready.

status: draft. Переход на ready — в шаге 5 после adversarial review.
```

### Выход

`FS/TKSC-001-handoff.md`:
- `status: draft`.
- 8 разделов по шаблону выше.
- Embedded Mermaid C4 диаграмма.
- Раздел 6.3 (Блокирующие) пуст.
- Signoff заполнен.
