# Analyst Workflow

> ℹ️ **Info — Контекст**
> Полный workflow enterprise-аналитика (системного / бизнес / solution) при работе с AI coding agents. Общие концепции, security и чеклисты — в [AI Agents](./AI%20Agents.md). Vendor-specific детали (subagents, MCP к Confluence / Jira / Bitbucket) — в [OpenCode Guide](./OpenCode%20Guide.md). Workflow разработчика — в [Developer Workflow](./Developer%20Workflow.md). Шаблон ADR — `[Decision Record](./Decision%20Record.md)`.

Раздел про то, как enterprise-аналитику использовать AI-агента в ежедневной работе: от приёма бизнес-запроса до передачи постановки в разработку и сопровождения реализации.

---

## 1. Принципы

Базовые принципы работы с AI-агентом (Plan first / контекст важнее промта / ответственность на человеке) — **[AI Agents](./AI%20Agents.md) §3** + **§5**. Применительно к аналитику:

- **Агент структурирует, не решает.** Он помогает упорядочить разрозненный вход, найти противоречия, подготовить черновик. Бизнес-решения не принимает.
- **Факт ≠ предположение.** Любое утверждение агента о том, как работает система или процесс, — гипотеза, пока не подтверждено первоисточником (код, регламент, владелец системы).
- **Бизнес-проблема и решение разделены.** Если бизнес говорит «нам нужно ускорить процесс», это требование. «Сделать это через Kafka» — решение, и его принимает архитектор.
- **Терминология — корпоративная.** Если в банке «Account» означает не то, что в учебниках, агент не догадается. Подсасывать глоссарий через `AGENTS.md` проекта или явно в промте.

## 2. Разделение ответственности

| Делает агент | Делает аналитик |
|---|---|
| Структурирует разрозненный вход (письма, тикеты, фрагменты Confluence) | Получает запрос от бизнеса, фиксирует контекст |
| Готовит черновик BPMN AS-IS / TO-BE | Согласует с владельцами процессов |
| Готовит user stories и acceptance criteria по описанию | Утверждает финальные формулировки |
| Сравнивает версии OpenAPI, готовит data mapping | Проверяет факты у владельцев систем |
| Готовит вопросы к бизнесу / архитектору / смежным командам | Задаёт вопросы и фиксирует ответы |
| Делает ревью полноты артефактов (открытые вопросы, edge cases) | Принимает решение об open questions |
| Готовит черновик ADR / release notes / bug report | Отвечает за финал, согласует, подписывает |

Агент **не делает**: согласование с бизнесом, принятие решения о scope, утверждение требований, выбор архитектуры, mediation между командами.

## 3. Базовый цикл (13 шагов)

```
 1. Запрос          — бизнес-инициатива, тикет, проблема от поддержки.
 2. Первичный контекст — Confluence, Jira, регламенты, прошлые ADR.
 3. AS-IS           — реконструкция процесса по источникам.
 4. Проблема и цели — что болит, измеримые KPI.
 5. TO-BE           — варианты целевого процесса.
 6. Уточнения       — открытые вопросы → бизнес.
 7. Системный ландшафт — затронутые системы, границы ответственности.
 8. Интеграции и API — что меняется в контрактах и событиях.
 9. ФТ + НФТ        — функциональные и нефункциональные требования.
10. Acceptance criteria — Gherkin / table-form.
11. Согласование    — бизнес → архитектор → dev → QA.
12. Сопровождение   — answers, scope control, изменения.
13. Документация    — обновление Confluence, release notes, traceability.
```

## 4. Перед началом задачи (входной артефакт)

Что аналитик должен дать агенту в первом сообщении:

- **Бизнес-проблема**: что болит, в одном-двух предложениях. С болью клиента / комплаенса / операционной модели, не «надо сделать сервис».
- **Цель и измеримый эффект**: SLA, конверсия, % одобренных, time-to-onboard, снижение FTE.
- **Заинтересованные стороны**: владелец продукта, бизнес-владелец процесса, архитектор, смежные команды, compliance, риски, security.
- **Ссылки на источники**: тикет (Jira), требование (Confluence), BPMN, OpenAPI / WSDL, регламент, ADR, политики безопасности.
- **Ограничения**: regulatory (152-ФЗ, 115-ФЗ, AML, KYC), временные (релизный поезд, заморозка прод), ресурсные (бюджет, доступные команды).
- **Известные системы и интеграции**: что уже знаем — front, middleware, бекенды, событийная шина, DWH.
- **Что согласовано / что пока неизвестно**: явный список, чтобы агент не «угадал» вместо вас.
- **Формат ожидаемого результата**: «postановка по корпоративному шаблону FRD», «BPMN + user stories + AC», «diff OpenAPI + список breaking changes».

Чем лучше входной артефакт, тем меньше итераций. «Напиши требования по PROJ-1234» приведёт к красивому, но придуманному документу.

## 5. Анализ бизнес-требований

Агент полезен для:

- **Структурирования разрозненного запроса**: набор писем + тикет + видеозвонок с бизнесом → единая problem statement, цели, эффект, стейкхолдеры.
- **Разделения проблемы и решения**: «нам нужен сервис X» — это решение; ниже него лежит реальная проблема (часто «процесс занимает 3 дня»).
- **Поиска противоречий**: бизнес говорит одно, регламент — другое, текущая система — третье.
- **Списка уточняющих вопросов**: 5–15 вопросов, ответы на которые закроют 80% неопределённости.
- **Резюме для согласования**: 1-страничный summary с проблемой, целями, scope, рисками, open questions — для встречи с владельцем продукта.

Промт — §21.2.

Антипаттерн: «обогащать» бизнес-запрос за бизнес. Если он сказал что-то расплывчато — это вопрос к бизнесу, а не задача для агента.

## 6. AS-IS — анализ текущего процесса

Источники реконструкции AS-IS:

- BPMN-диаграмма из Confluence (часто устарела — проверять);
- регламент / процедура;
- Jira-тикеты с текущими багами и обходами;
- переписка / чаты с поддержкой;
- логи / выгрузки из БД (обезличенные);
- интервью с владельцами процесса.

Что просим у агента:

- Сводка процесса: участники, шаги, gateways, события, артефакты.
- BPMN AS-IS черновик (PlantUML / Mermaid / XML BPMN).
- **Ручные операции** и обходные пути (где люди вмешиваются между системами).
- Узкие места: ожидание подтверждения, дубли, двойной ввод, manual reconciliation.
- Неявные бизнес-правила: исключения для VIP-клиентов, регионов, продуктовых линий.
- Чёткое разделение **факта от желаемого**: что реально работает сегодня vs что должно работать «по идее».

Промт — §21.3.

> ⚠️ **Warning — Старая документация ≠ факт**
> 70% корпоративных BPMN в Confluence устарели на 1–3 года. Перед фиксацией AS-IS — обязательная проверка с владельцем процесса.

## 7. TO-BE — проектирование целевого процесса

Что агент готовит:

- **2–3 варианта** целевого процесса с pros/cons. Без рекомендации — выбор за бизнесом / архитектором.
- BPMN TO-BE: роли (lanes), события, действия, gateways, артефакты.
- **Happy path** (основной сценарий) + **alternative flows** (легальные альтернативы) + **exception flows** (ошибки, отказы, таймауты, retry).
- Проверка полноты: нет ли висячих веток, есть ли явный «end», обработаны ли таймауты, что происходит при сбое внешней системы.
- Список вопросов к бизнесу (правила) и к архитектору (реализуемость).

Промт — §21.4.

Антипаттерн: «нарисовать TO-BE и согласовать после». TO-BE без открытых вопросов = TO-BE с непроявленными конфликтами.

## 8. Системный анализ

Цель — понять, какие системы затронуты и как они взаимодействуют.

- **Какие системы затронуты**: front, middleware, BPM, мастер-системы, шина (Kafka / IBM MQ), batch-jobs, DWH, отчётность, аудит.
- **Системный ландшафт**: опираемся на C4 Context архитектора. Если C4 нет — это вопрос к архитектору, а не задача для агента.
- **Границы ответственности (Bounded Context)**: что входит в зону ответственности каждой системы. Это не данные, а бизнес-смысл.
- **Точки интеграции**: REST / SOAP / async event / shared DB / file-drop / batch-export.
- **I/O анализ**: входные данные (откуда, формат, частота), выходные (куда, формат, гарантии доставки).
- **Side effects**: audit-логи, нотификации (SMS / push / email), материализация в DWH, события в шину, обновления связанных entities.
- **Зависимости**: что должно произойти до этой системы, что — после.

> ⚠️ **Warning — Не придумывать поведение системы**
> Самая частая ошибка: «думаю, эта система делает X». Без подтверждения от владельца / кода / регламента — фиксируем как **гипотезу**, не как факт.

Промт — §21.5 (адаптируется под системный анализ).

## 9. API и интеграции

Агент полезен для:

- **Анализ Swagger / OpenAPI**: паспорт эндпоинтов, обязательные / опциональные / условные поля, error codes.
- **Описание REST / SOAP / async**: payload, headers, идемпотентность, retry policy, timeout.
- **Kafka events**: схема payload, headers, ключ партицирования, retention, при каких бизнес-условиях публикуется, кто consumer'ы.
- **Проверка обязательности полей**: какие реально required, а какие «формально optional, но без них ничего не работает».
- **Diff API**: добавленные / удалённые эндпоинты, breaking changes (тип, обязательность, путь, статус-код, формат), backward-compatible.
- **Примеры запросов / ответов**: один валидный, один с ошибкой 4xx, один 5xx, один с бизнес-ошибкой.
- **Описание ошибок**: коды, формат `ProblemDetail` / корпоративный, когда какая ошибка возвращается.
- **Backward compatibility**: что произойдёт со старыми consumer'ами после изменения.
- **Вопросы к владельцам смежных систем**: что хотим знать перед интеграцией.

Промт — §21.5.

## 10. Работа с данными

Агент полезен для:

- **Структуры данных**: какие entity, атрибуты, связи, identifiers.
- **Data mapping**: две схемы (например, миграция / интеграция) → таблица соответствия с пометками `1:1`, `1:N`, `transform`, `derived`, `not mapped`.
- **Источники и потребители**: для каждого атрибута — кто master, кто consumer.
- **Required / optional / calculated** поля: что обязательно, что опционально, что вычисляется.
- **Расхождения в терминах**: «Account» в системе A ≠ «Account» в системе B. Глоссарий с явным определением для каждой системы.
- **Mapping таблица**: формат, где левая колонка — source, правая — target, посередине — правило преобразования.
- **Validation rules**: длина, формат (regex), допустимые значения, кросс-полевая валидация (если поле X = «Y», то поле Z обязательно).
- **DQ checks**: uniqueness (нет дубликатов), completeness (нет null в обязательных), referential integrity (FK существуют), reasonableness (значения в разумных диапазонах).

Промт — §21.6.

## 11. Подготовка требований

### 11.1 Виды требований

- **Бизнес-требования**: что нужно бизнесу (от лица клиента / процесса).
- **Функциональные требования (ФТ)**: что система должна делать.
- **Нефункциональные требования (НФТ)**: SLA, throughput, availability, RPO/RTO, security, observability, compliance.
- **Системные требования**: технические ограничения и интерфейсы между системами.
- **Ограничения**: что мы НЕ можем менять (regulatory, legacy contracts, infra).
- **Допущения (assumptions)**: что мы принимаем как данное, не проверяя.
- **Acceptance criteria (AC)**: проверяемые условия приёмки в формате Given/When/Then или table-form.
- **Definition of Ready (DoR)**: критерии, по которым задача готова к разработке (см. §25).
- **Definition of Done (DoD)**: критерии, по которым задача считается завершённой.
- **Open questions**: неотвеченные вопросы, требующие решения.
- **Out of scope**: явно перечисленное «что не делаем в этой задаче».

### 11.2 Разница между понятиями

| Тип | Что это | Пример |
|---|---|---|
| **Требование** | Что система должна делать | «Скоринг должен возвращаться < 5 секунд» |
| **Решение** | Как это реализовано | «Используем кеш Redis с TTL 1 мин» |
| **Гипотеза** | Предположение, не подтверждённое | «80% клиентов проходят с первой попытки» |
| **Ограничение** | Что мы не можем менять | «Нельзя менять формат outbound message в SAP» |
| **Бизнес-правило** | Логика домена | «Скоринг не выдаётся клиентам < 18 лет» |
| **Acceptance criteria** | Как проверить требование | «Given валидный клиент 25 лет, When запрошен скоринг, Then ответ < 5 сек» |

В постановке эти типы должны быть **явно разделены**. Перемешанный текст «нам нужен сервис скоринга на Postgres со SLA 5 секунд для клиентов от 18 лет, потому что 80% проходят» — нечитаемая постановка, в которой невозможно отличить, что менять, а что — данность.

## 12. Декомпозиция задач

Агент полезен для:

- **Initiative → epic → story → task** (по корпоративной иерархии Jira).
- **User stories** в формате `As a <role>, I want <action> so that <benefit>`.
- **Технические задачи**: миграция БД, изменения OpenAPI, события Kafka, инфра.
- **Зависимости**: «Story A блокирует Story B», «Task X должен быть готов до релиза Y».
- **Блокеры**: что мешает начать, кто разблокирует.
- **Задачи по командам**: backend / frontend / mobile / QA / DevOps / Data — кто что делает.
- **Проверка непротиворечивости**: задачи не пересекаются, не конфликтуют, нет «висячих» story без owner'а.

Промт — §21.7.

## 13. Постановка для разработки

Структура качественной постановки:

```
1. Цель                — одна фраза, в чём смысл задачи.
2. Бизнес-контекст     — почему делаем, ссылка на инициативу / тикет.
3. AS-IS поведение     — как сейчас.
4. TO-BE поведение     — как должно быть.
5. Затронутые системы  — список + границы изменений.
6. Изменения API       — endpoints / payload / events / deprecations.
7. Изменения данных    — БД миграции, новые поля, переходный период.
8. Бизнес-правила      — явный список, нумерованный.
9. Сценарии            — happy / alt / exception, с примерами.
10. Ограничения        — regulatory / NFR / технические.
11. Acceptance criteria — Gherkin или table-form.
12. Тестовые данные    — обезличенные примеры.
13. Open questions     — что не решено.
14. Ссылки             — Confluence / Jira / OpenAPI / ADR / BPMN.
```

Промт — §21.11. Чеклист готовности — §25.

## 14. Проверка требований (review)

Агент полезен для self-review постановки до согласования:

- **Неоднозначности**: «быстро», «просто», «при необходимости», «в большинстве случаев» — найти и переформулировать.
- **Противоречия**: AC #3 говорит одно, бизнес-правило #5 — другое.
- **Пропущенные сценарии**: что если внешняя система недоступна? Что при таймауте? Что при дубликате?
- **Полнота AC**: каждое требование покрыто хотя бы одним AC.
- **Соответствие бизнес-цели**: AC реально проверяют достижение KPI.
- **Согласованность терминов**: один и тот же объект называется одинаково во всём документе.
- **Edge cases**: null, пустые коллекции, граничные значения, конкурентные операции.
- **Testability**: можно ли написать тест по этому AC? Если нет — переформулировать.

Промт — §21.9.

## 15. Взаимодействие с архитектором

Агент полезен для:

- **Подготовки вопросов**: список явных вопросов к встрече с архитектором (а не «что ты думаешь?»).
- **Описания вариантов решения**: 2–3 подхода с pros/cons. Решение принимает архитектор.
- **Impact analysis**: какие системы / контракты / consumer'ы затронуты.
- **ADR draft**: по шаблону `[Decision Record](./Decision%20Record.md)`. Финал утверждает архитектор.
- **Trade-offs**: явное «что мы теряем» в каждом варианте.
- **C4 Context / Container**: PlantUML черновик для обсуждения.
- **Архитектурные ограничения**: фиксируем как ограничения в постановке.
- **Перевод arch-решения в требования**: после утверждения — разбиваем в задачи.

Антипаттерн: «агент сказал, что вариант X лучше → ставим X в постановку». Архитектурное решение принимает человек.

## 16. Взаимодействие с разработчиками

Агент полезен для:

- **Подготовки понятной постановки** (см. §13).
- **Объяснения бизнес-логики**: почему именно такие правила, какие исключения, какие приоритеты.
- **Подготовки примеров**: payload, тестовые данные, сценарии.
- **Разбор спорных сценариев**: разработчик задаёт вопрос → агент помогает аналитику собрать ответ из источников.
- **Анализ вопросов от dev'ов**: «вопрос разработчика → пробежать по постановке → найти, что неоднозначно → уточнить».
- **Обновление требований после технических уточнений**: фиксировать в постановке, а не в чате.
- **Фиксация договорённостей**: «договорились вот так, в Confluence обновлено, в Jira комментарий».

Промт — §21.11.

## 17. Взаимодействие с QA

Агент полезен для:

- **Тест-кейсов**: positive / negative / edge — таблица или Gherkin.
- **Тестовых данных**: обезличенные примеры по сценариям.
- **Связки требования ↔ тесты (traceability)**: каждое требование покрыто хотя бы одним тест-кейсом.
- **Покрытия AC**: таблица «AC → тест-кейс».
- **Чек-листа manual testing**: для сценариев, где автотест дорог.
- **Регрессионных областей**: что нужно прогнать после фикса.

Промт — §21.8.

Антипаттерн: «отдать тест-кейсы в QA без обсуждения». QA знает edge cases, которых аналитик не видит.

## 18. Сопровождение реализации

После передачи задачи в разработку аналитик не отключается. Агент помогает:

- **Анализ вопросов команды**: каждый вопрос → пройти по постановке → дать ответ или зафиксировать gap.
- **Актуализация требований**: уточнения после технических находок фиксируем в постановке.
- **Scope creep control**: новый запрос «давай заодно ещё это» → отдельный тикет, не в текущую задачу.
- **Соответствие реализации постановке**: при ревью dev-демо проверять, что AC выполнены.
- **Release notes**: краткое описание для бизнеса и пользователей.
- **Документация**: обновление Confluence / API spec / BPMN после релиза.

Антипаттерн: «постановка ушла в dev → задача аналитика закрыта». Нет — задача закрыта после релиза и обновления документации.

## 19. Инциденты и баги

Сценарий работы:

1. **Описание проблемы** от поддержки / клиента / мониторинга.
2. **Симптомы**: что именно происходит, на каком шаге, у кого.
3. **Анализ**: логи, примеры запросов, выгрузки данных, скрины.
4. **Фактический сценарий**: что реально происходит шаг за шагом.
5. **Ожидаемое поведение**: как должно быть по требованиям.
6. **Классификация**:
   - **Bug**: реализация не соответствует требованиям → fix в коде.
   - **Gap requirements**: требование не было сформулировано → доформулировать + fix.
   - **Изменение бизнес-ожиданий**: бизнес поменял мнение → новая инициатива, не bug.
7. **Bug report**: симптом / шаги воспроизведения / ожидаемое / фактическое / impact / приоритет.
8. **Требования к исправлению**: что должно измениться, AC для проверки.
9. **Impact на смежные сценарии**: что ещё могло сломаться, что нужно регрессировать.

## 20. Legacy-системы

В legacy-окружении (старые монолиты, generated stubs, заброшенные системы) аналитика особенно осторожна.

- **Документация ≠ факт**: проверять каждое утверждение из старой Confluence через код / выгрузку / интервью.
- **Фактическое поведение** фиксируется отдельно от «как должно быть».
- **Неявные бизнес-правила** в legacy — норма: исключения для конкретного партнёра, обходы для регуляторного дедлайна, костыли после инцидента 2019 года.
- **Устаревшие термины**: вести таблицу «старый термин → новый термин (если есть)». Не переименовывать в legacy без необходимости.
- **Backward compatibility**: ломать публичные контракты legacy = ломать прод. Расширять — да, удалять — только по deprecation-плану.
- **Минимизация риска**: характеристические тесты для текущего поведения до правок.
- **Документирование найденного**: каждая работа с legacy обогащает knowledge base. Записывать в `docs/<system>/notes.md` или `AGENTS.md` модуля.

## 21. Промты — хорошие и плохие

### 21.1 Плохой промт

```
Напиши требования по PROJ-1234.
```

Что не так: нет бизнес-проблемы, нет цели, нет ограничений, нет источников. Агент будет выдумывать.

### 21.2 Рекомендуемый базовый промт для enterprise-аналитика

```
Контекст:
- Бизнес-проблема: <одно-два предложения, что болит>.
- Цель и измеримый эффект: <KPI>.
- Стейкхолдеры: <product owner, бизнес, архитектор, смежные команды>.
- Ссылки: <Jira>, <Confluence>, <BPMN>, <OpenAPI>, <регламенты>.
- Известные системы и интеграции: <список>.
- Ограничения: <regulatory / NFR / временные / ресурсные>.
- Что согласовано: <список>.
- Что неизвестно: <список>.
- Глоссарий: <ключевые термины с определениями для нашего домена>.

Задача:
Сейчас НЕ финализируй документ. Сделай:
1. Резюме бизнес-проблемы и цели в 1 параграфе.
2. Список затронутых систем с границами изменений.
3. Открытые вопросы, ранжированные по критичности (что блокирует, что можно решить позже).
4. Список рисков и ограничений.
5. План аналитической работы: что нужно сделать до постановки в разработку.

Формат: markdown с разделами 1–5. Не выдавай предположения за факты — гипотезы помечай явно.
```

### 21.3 Анализ AS-IS

```
По следующим источникам реконструируй процесс AS-IS:
- регламент: <вставка / ссылка>;
- BPMN из Confluence: <вставка / ссылка>;
- 3 тикета поддержки: <ссылки>;
- выгрузка из БД (обезличенная): <вставка>.

Верни:
1. Сводку процесса: участники, шаги, gateways, артефакты.
2. BPMN AS-IS в PlantUML / Mermaid.
3. Ручные операции и обходные пути.
4. Узкие места и проблемные точки.
5. Неявные бизнес-правила (исключения, особые случаи).
6. Расхождения между регламентом и тем, что описывают тикеты.
7. Список фактов vs предположений.

Не сглаживай противоречия. Не предлагай TO-BE.
```

### 21.4 Проектирование TO-BE

```
По AS-IS (см. <ссылка>) и целям (<ссылка>) предложи 2–3 варианта TO-BE:

Для каждого варианта:
1. Краткое описание.
2. BPMN TO-BE в PlantUML / Mermaid: lanes, события, gateways, exception flows.
3. Pros / Cons.
4. Trade-offs (что теряем).
5. Влияние на смежные системы.
6. Требования, которые этот вариант создаёт.

Не давай рекомендацию — выбор за бизнесом и архитектором.
В каждом варианте обязательно: happy path, alternative flow, exception flow.
В конце — единый список вопросов к бизнесу и архитектору.
```

### 21.5 API и интеграция

```
Сравни api/v1/payments-openapi.yaml и api/v2/payments-openapi.yaml.
Верни:
1. Добавленные / удалённые эндпоинты.
2. Breaking changes по полям (тип, required, формат, enum).
3. Backward-compatible изменения.
4. Для каждого breaking change — кто из downstream-консьюмеров затронут.
5. Примеры запросов / ответов: 1 успех, 1 валидационная ошибка 4xx, 1 серверная 5xx.
6. Список вопросов к владельцу API <команда / контакт>.

Не выдумывай consumer'ов — если не указано, помечай как «требует уточнения у владельца API».
```

### 21.6 Data mapping

```
Сравни две схемы:
- source: <вставка JSON Schema / DDL / OpenAPI component>;
- target: <вставка>.

Верни таблицу:
| target.field | source.field | rule | notes |
|---|---|---|---|
с пометками: 1:1, transform (с формулой), derived (с источниками), default, not mapped, conditional.

Дополнительно:
1. Список target-полей без source — где брать?
2. Список source-полей без target — теряются?
3. Расхождения в типах / форматах / enum.
4. Поля с разной семантикой при одинаковом названии.
5. Validation rules для каждого target-поля.
```

### 21.7 User story

```
Из требования <ссылка / вставка> сформируй user stories.

Формат: As a <role>, I want <action>, so that <benefit>.

Для каждой:
- 3–5 acceptance criteria в Gherkin (Given / When / Then);
- зависимости от других stories;
- открытые вопросы;
- какая команда выполняет (backend / frontend / mobile / QA / DevOps / Data);
- предполагаемый размер (XS / S / M / L) с обоснованием;
- ссылки на затронутые модули и контракты.

Не объединяй stories разной природы. Если story больше L — предлагай декомпозицию.
```

### 21.8 Acceptance criteria

```
Для требования <ссылка / вставка> подготовь acceptance criteria.

Формат — Gherkin (Given / When / Then) или table-form для сценариев с большим перебором.

Покрой:
1. Happy path.
2. Alternative flows (легальные альтернативы).
3. Exception flows (ошибки внешних систем, таймауты, дубликаты).
4. Edge cases (null, пустые коллекции, границы числовых диапазонов, конкурентные операции).
5. Negative-сценарии (валидация, отказ доступа, бизнес-запрет).

Каждый AC должен быть проверяемым автотестом или явно помечен как «manual only».
```

### 21.9 Review требований

```
Прочитай постановку <ссылка / вставка> и сделай ревью.

Верни:
1. Неоднозначные формулировки («быстро», «просто», «при необходимости») с предложением переформулировки.
2. Противоречия между разделами.
3. Пропущенные сценарии (что если внешняя система недоступна, что при таймауте, что при дубликате).
4. AC, которые не проверяемы или не покрывают требование.
5. Несогласованность терминов (один объект под разными именами).
6. Edge cases, не отражённые в AC.
7. Требования без owner'а / без ответственной команды.
8. Open questions, не зафиксированные явно.

В конце — оценка готовности к передаче в разработку: ready / needs work / blocked, с обоснованием.
```

### 21.10 Подготовка вопросов бизнесу

```
По проблеме <вставка> и AS-IS <ссылка> подготовь список вопросов к бизнесу для встречи 30 минут.

Сгруппируй:
1. Бизнес-цель и приоритеты (что важнее: скорость, точность, стоимость).
2. Бизнес-правила и исключения (особые клиенты / регионы / продукты).
3. Граничные сценарии (что делать при ошибках, отказах, дубликатах).
4. Нефункциональные ожидания (SLA, объёмы, доступность).
5. Out of scope (что точно не делаем).

Для каждого вопроса — почему важен. Не задавай вопросов, на которые ответ есть в источниках.
Расставь приоритеты: блокирующие → желательные → справочные.
```

### 21.11 Постановка задачи разработке

```
По требованиям <ссылка> и AC <ссылка> подготовь постановку для разработки по нашему шаблону.

Структура:
1. Цель.
2. Бизнес-контекст.
3. AS-IS поведение.
4. TO-BE поведение.
5. Затронутые системы.
6. Изменения API (с diff OpenAPI).
7. Изменения данных (миграции, новые поля).
8. Бизнес-правила (явный нумерованный список).
9. Сценарии (happy / alt / exception).
10. Ограничения.
11. Acceptance criteria.
12. Тестовые данные (обезличенные).
13. Open questions.
14. Ссылки.

Формат — markdown по нашему шаблону FRD. Не добавляй технические решения (выбор архитектуры — за разработкой).
```

## 22. Безопасность и контроль

> ⚠️ **Warning — Применение к workflow аналитика**
> Полные правила — **[AI Agents](./AI%20Agents.md) §13** (что нельзя отдавать модели, PII, провайдеры, MCP, audit-trail) и **§13.2** (PII и коммерческая тайна). Analyst-specific акценты:
>
> - **Обезличивание**: данные клиентов в примерах — placeholder'ы (`Иванов И.И.`, `1234 567890`), не реальные. Маскирование выполняется ДО передачи в context window.
> - **Закрытые регламенты и NDA-документы** — только через AlfaGen, не публичные агенты.
> - **Не выдавать предположение агента за согласованное решение.** Если в постановке написано «по согласованию с X» — должна быть ссылка на запись согласования (тикет / комментарий / протокол).
> - **MCP Confluence/Jira** (`mcp-atlassian`) — только чтение, не запись правил без согласования (см. [AI Agents](./AI%20Agents.md) §13.6).
>
> Финальная подпись на постановке — ответственность аналитика. См. [AI Agents](./AI%20Agents.md) §5.

## 23. Антипаттерны

Общие антипаттерны (без рамок задачи, использование агента для обхода согласований, передача PII / коммерческой тайны во внешний контур, vibe architecting) — **[AI Agents](./AI%20Agents.md) §14**. Analyst-specific:

- Смешивание бизнес-проблемы и технического решения в одном предложении.
- Не фиксировать open questions явно. Молчаливые предположения = инциденты после релиза.
- Согласовать постановку с бизнесом, не показав разработке и QA.
- Игнорировать exception flows ради «happy path выглядит хорошо».
- Copy-paste текста агента без адаптации под корпоративный шаблон и глоссарий.
- Не обновлять документацию после изменений в реализации. Confluence расходится с прод за 1–2 спринта.
- Принимать «красиво звучащие» формулировки без проверки. Особенно опасно в нефункциональных требованиях.
- Не отделять факт от предположения. AS-IS превращается в «как мы думаем оно работает».

## 24. Минимальный чеклист готовности аналитической постановки (Definition of Ready)

- [ ] Бизнес-проблема сформулирована (не «нужен сервис», а «процесс длится 3 дня вместо 1»).
- [ ] Цель и измеримый KPI зафиксированы.
- [ ] Стейкхолдеры перечислены с ролями (product owner, бизнес-владелец, архитектор, смежные команды).
- [ ] Ссылки на источники: Jira, Confluence, BPMN, OpenAPI, регламенты, ADR.
- [ ] AS-IS зафиксирован и проверен с владельцем процесса.
- [ ] TO-BE согласован с бизнесом и архитектором.
- [ ] Затронутые системы перечислены, границы изменений описаны.
- [ ] Изменения API задокументированы (diff OpenAPI / Kafka events / breaking changes).
- [ ] Изменения данных описаны (миграции, новые поля, переходный период).
- [ ] Бизнес-правила перечислены явно и нумерованы.
- [ ] Сценарии описаны: happy / alt / exception.
- [ ] Acceptance criteria покрывают все требования и проверяемы.
- [ ] НФТ описаны (SLA, throughput, availability, security, observability, compliance).
- [ ] Ограничения и допущения зафиксированы.
- [ ] Out of scope явно указан.
- [ ] Open questions перечислены и ранжированы по критичности.
- [ ] Тестовые данные подготовлены (обезличенные).
- [ ] Постановка прошла self-review агентом и читалась автором свежим взглядом.
