## 3. Базовый паттерн

Любая нетривиальная задача с AI-агентом проходит 7 шагов. Пропуск шагов — антипаттерн (§14).

```
1. Анализ        — изучить, что есть, как работает, где менять, какие риски.
2. План          — описать минимальные правки и порядок.
3. Подтверждение — пользователь читает план и валидирует.
4. Маленькие правки — точечные изменения с explicit-scope.
5. Diff review   — пользователь читает diff каждой правки.
6. Тесты         — вручную, прежде чем что-то закоммитить.
7. Ручное принятие — `git commit` от человека, не автоматически.
```

Шаги 1–3 — в `Plan`-режиме (read-only, для анализа), шаги 4–6 — в `Build` (с edit/bash, для реализации). В OpenCode переключение между ними — см. [OpenCode Guide](./OpenCode%20Guide.md).

### 3.1 Полный цикл vs сокращённый

| Критерий | Полный 7-шаговый цикл | Сокращённый путь |
|---|---|---|
| Логика | меняет логику | косметика (rename, форматирование, опечатки, комментарии) |
| Объём | > 1 файла или > 30 строк | ≤ 1 файл, ≤ 30 строк |
| Логические развилки | есть | нет |
| Публичный API / контракты | затронуты | не затронуты |
| Категория | security / crypto / migrations / payments / IaC | вне этих категорий |
| Согласование | архитектурное / security / DBA | не требуется |
| Шаги | §3 (1–7) | Build → diff-review (§13.3) → тесты → коммит |

**При сомнении — полный цикл.** Любое одно совпадение со столбцом «полный цикл» поднимает задачу до полного цикла.

### 3.2 Большие и сложные задачи

Полного 7-шагового цикла недостаточно, если задача архитектурная или меняет несколько систем. Тогда нужны **планирующие артефакты** — готовятся ДО первой `Build`-сессии и существуют в репо как persistent-контекст между сессиями.

**Признаки большой задачи** (любое одно — переходим в этот режим):

- затрагивает > 1 сервис / контракт / схему данных;
- меняет публичный API или интеграционные контракты;
- migration / breaking change / новая зависимость на инфраструктурный уровень;
- работа > 1 спринта или > 1 разработчика;
- architecture-level decision (не bug fix / не фича в одном модуле);
- неочевидный scope, требующий сначала анализа.

**Какие артефакты готовятся:**

| Артефакт | Когда нужен | Шаблон / детали |
|---|---|---|
| ADR (Architecture Decision Record) | architecture-level решения, breaking changes | [Decision Record](./Decision%20Record.md), [Architect Workflow](./Architect%20Workflow.md) §14 |
| Impact analysis | задача затрагивает > 1 сервис / команду | [Architect Workflow](./Architect%20Workflow.md) §9 |
| C4-диаграммы (Context + Container) | новый сервис / большая интеграция | [Architect Workflow](./Architect%20Workflow.md) §15 |
| Migration plan + rollback | breaking change, schema migration | часть ADR ([Architect Workflow](./Architect%20Workflow.md) §14) |
| Spec / постановка с AC | любая нетривиальная задача | [Analyst Workflow](./Analyst%20Workflow.md) §15 |
| Roadmap / декомпозиция в эпики и тикеты | работа > 1 спринта | пример: AI Agents — Audit Roadmap (внутренний rollout-план, не публикуется) |

**AI-специфичная дисциплина:**

1. **Декомпозируй ДО начала AI-сессии**, не во время. Большая задача в одной сессии = переполненный контекст и расхождение с планом.
2. **Один артефакт = одна сессия.** ADR-черновик, migration plan, реализация под-задачи — каждое в своей сессии с узким scope. Типовой цикл: `Plan` (read-only обсуждение варианта) → переключение в `Build` для записи файла (`docs/decisions/<id>.md` и т.п.) → коммит → закрытие сессии. Не смешивать несколько артефактов в одной сессии.
3. **Архитектура — за человеком, не за агентом.** Агент пишет черновик ADR, решение принимает архитектор. Антипаттерн «vibe architecting» — §14.
4. **Промежуточные артефакты в репо** (`docs/decisions/`, `docs/architecture/`, корневой `AGENTS.md` / `CLAUDE.md`) — это persistent-контекст для следующих сессий. Без них агент каждый раз начинает с нуля.
5. **Issue-quality перед делегированием.** AC, список затронутых файлов, тесты, ограничения. Если задача формулируется размыто — это сигнал на ADR/spec, а не на `Build`-сессию.
6. **Cross-session continuity.** Привязывай одну инициативу к одной git-ветке и используй механизм возобновления сессии своего инструмента (синтаксис resume — в гайде по инструменту §2). Audit-trail коммитов: label `ai-assisted` + краткий промт в описании PR.

**Чеклист «готов к большой AI-сессии»:**

- [ ] ADR — драфт или утверждён.
- [ ] Impact analysis сделан, затронутые сервисы и команды известны.
- [ ] Декомпозиция в ≤ 5 атомарных под-задач, каждая ≤ одна Plan→Build сессия.
- [ ] Acceptance criteria каждой под-задачи письменно.
- [ ] Migration / rollback план готов (если breaking change).
- [ ] Source-материалы (тикеты, AC, контракты, ADR) в едином месте, ссылки готовы для промта.
- [ ] Согласование архитектора / DBA / security получено (если применимо).
- [ ] `AGENTS.md` / `CLAUDE.md` проекта актуален и содержит запреты.

**Связанные разделы:** §3.1 (тривиальные vs нетривиальные), §6.1 (project rules как persistent-контекст), §14 (антипаттерны).

## 4. Когда AI использовать НЕЛЬЗЯ

AI-агент — инструмент, не замена эксперта. Ниже — кейсы, где использование AI **запрещено** или **допустимо только с обязательным дополнительным контролем**.

### 4.1 Категорически запрещено

- **Генерация ключей, паролей, токенов, сертификатов** — модели подвержены коллизиям и могут логировать вход; используются криптостойкие генераторы из стандартных библиотек.
- **Действия в production-инфраструктуре** (миграции БД, изменения конфигов, рестарт сервисов, выкладки) без явного approval'а владельца сервиса.
- **Принятие архитектурных, бизнес- или юридических решений от имени AI** — модель может предложить вариант, но решение принимает человек с зафиксированной ответственностью.
- **(QA)** Финальный вердикт о готовности релиза.
- **(QA)** Замена ручного исследовательского тестирования (exploratory) — AI пропускает usability-проблемы.
### 4.2 Допустимо только с обязательным human-ревью и независимой валидацией

- **Криптография, аутентификация, авторизация, ACL** — обязательное security-ревью перед merge.
- **Миграции production-БД** — обязательный dry-run на копии прод-данных и review DBA.
- **Production-инциденты в режиме «горит»** — AI допустим как ассистент анализа, но команда восстановления исполняется человеком пошагово, с подтверждением каждого шага.
- **Legacy-системы без тестов** — AI пишет правку только после того, как человек убедился, что побочные эффекты предсказуемы.

> Признаки прерывания сессии и порядок действий — §12.3.

## 5. Ответственность

AI-агент **не является субъектом ответственности**. Ответственность за результат всегда несёт человек.

| Действие                                      | Кто отвечает                           |
| --------------------------------------------- | -------------------------------------- |
| Содержимое коммита, MR, PR                    | Автор коммита (человек)                |
| Архитектурное решение                         | Архитектор / автор ADR                 |
| Бизнес-постановка                             | Системный аналитик / владелец продукта |
| Тест-кейс и его валидность                    | QA                                     |
| Публикация / merge в защищённую ветку         | Ревьюер и автор                        |
| Подключение нового MCP                        | Владелец сервиса + security            |


**Label `ai-assisted` означает**, что AI участвовал в подготовке артефакта и ревьюер должен учитывать риск галлюцинаций. **Не означает**, что human-ревью или архитектурное / security / DBA-ревью можно сократить или ускорить.

## 6. Концепции

### 6.0 Сравнение

| Свойство             | Skill                                  | Subagent                                        | Command                        | MCP                                 |
| -------------------- | -------------------------------------- | ----------------------------------------------- | ------------------------------ | ----------------------------------- |
| Что это              | переиспользуемая «специализация»       | изолированный исполнитель со своими permissions | параметризованный промт-шаблон | внешний сервер с tools              |
| Когда применять      | повторяющаяся задача                   | нужен ограниченный исполнитель                  | один и тот же промт ≥10 раз    | нужен доступ к внешней системе      |
| Как вызывается       | автоматически при подходящем контексте | `@mention` или task-tool                        | `/<name>` слэш-команда         | через инструменты после подключения |
| Permissions          | наследует от агента                    | свои (изолированы)                              | наследует                      | свои + наследует                    |
| Основные риски       | широкая активация — ошибочный контекст | избыточные permissions                          | устаревший шаблон              | write в production, утечка          |


### 6.1 Project rules

Файл правил проекта, который агент читает при старте. Без него модель не знает domain-специфики.

| Инструмент  | Файл                            |
| ----------- | ------------------------------- |
| OpenCode    | `AGENTS.md` (создаётся `/init`) |
| Claude Code | `CLAUDE.md` (создаётся `/init`) |
| KiloCode    | `AGENTS.md`                     |

**MUST-have контент:**

- контекст проекта (стек, ключевые модули, внешние зависимости);
- конвенции (стиль, naming, коммиты, ветки);
- запреты (что НЕ менять, какие библиотеки/паттерны не использовать);
- куда писать тесты + минимальное покрытие;
- Definition of Done;
- контакты владельца / канал для вопросов;
- дата последнего ревью и следующего (квартально).

**Антипаттерны:** см. §14.

**Каркас** (tool-agnostic — структура одинакова для `AGENTS.md`, `CLAUDE.md`, `.cursorrules` и эквивалентов; имя файла и механизм загрузки — в гайде по инструменту §2):

```markdown
# <Имя проекта>

## Контекст
<Стек, ключевые модули, почему этот проект существует.>

## Конвенции
- стиль кода: <ссылка на гайд / linter>
- naming: <функции / переменные / файлы>
- коммиты: <Conventional Commits / прочее>
- ветки: <main / feature/<ticket>-<slug>>

## Запреты на изменение
- не редактируй `<dir>/` без согласования с <ответственный>
- не запускай продовые миграции
- не трогай `<file>`, он генерируется
- не меняй публичные API/контракты без ADR

## Что нельзя отдавать модели
- содержимое `<dir>/secrets/` (закрыто на уровне permissions инструмента, здесь — напоминание)
- ПДн клиентов: ФИО, паспорт, телефон, email, счета — без маскирования
- стек-трейсы из прод-логов с пользовательскими полями
- внутренние финансовые показатели и контракты
- полный список — [AI Agents](./AI%20Agents.md) §13.1

## Тесты
- где лежат: `<path>`
- паттерн: `<Arrange-Act-Assert / Given-When-Then>`
- запуск: `<command>`
- минимальное покрытие: <процент>

## Audit-trail
- AI-задействованные коммиты: label `ai-assisted` + 1–3 строки промта в PR
- для security/crypto/migrations/payments — полный лог сессии в Confluence/wiki
- инцидент применения AI — по процедуре [AI Agents](./AI%20Agents.md) §12.3

## Definition of Done
- [ ] тесты зелёные
- [ ] `git diff --staged` прошёл человеческое ревью
- [ ] PR имеет описание со ссылкой на задачу
- [ ] label `ai-assisted` если был AI-агент
- [ ] pre-commit ревью пройдено

## Контакты
- владелец сервиса: <person>
- канал: <slack / mattermost / teams>

## Ревью документа
- Последний ревью: <дата>
- Следующий ревью: <дата +квартал>
```

Vendor-specific дополнения (синтаксис, иерархия загрузки, особенности файлов поддиректорий) — в гайде по инструменту (§2).

## 7. Workflow по ролям

Все ролевые циклы — расширение общего 7-шагового паттерна (§3) под специфику. 

| Роль                  | Полный workflow (spoke)                | Цикл     | Промт-библиотека        | Минимум для старта                                                                    |
| --------------------- | -------------------------------------- | -------- | ----------------------- | ------------------------------------------------------------------------------------- |
| Разработчик           | [Developer Workflow](./Developer%20Workflow.md)                 | 10 шагов | spoke §14 (6 шаблонов)  | вход → `Plan` → принять план → `Build` атомарными diff'ами → тесты + diff-review → PR |
| Аналитик              | [Analyst Workflow](./Analyst%20Workflow.md)                   | 13 шагов | spoke §21 (12 шаблонов) | бизнес-запрос → AS-IS → TO-BE → ФТ/НФТ/AC → передача в разработку                     |
| Архитектор            | [Architect Workflow](./Architect%20Workflow.md)                 | 16 шагов | spoke §24 (12 шаблонов) | инициатива+NFR → AS-IS → 2–3 варианта TO-BE с impact-анализом → ADR → C4 + контракты  |
| QA                    | inline §7.1                            | 5 шагов  | §7.1.3                  | AC → покрытие → тест-план → черновики → дефекты с аудит-следом                        |

QA не имеет отдельного spoke-документа — цикл и промты для него приведены ниже inline.

### 7.1 QA

#### 7.1.1 Цикл (5 шагов)

1. Получить вход: AC, постановка, прод-логи / сценарии, тестовые данные.
2. `Plan`-режим: попросить агента предложить покрытие (позитивные / негативные / edge), отметить пробелы в AC.
3. Сформировать тест-план: unit / integration / e2e / нагрузочные / регрессионные.
4. Сгенерировать черновики тест-кейсов (Gherkin / чеклист / код); валидировать вручную — модель часто пропускает negative paths.
5. По итогам прогона — задокументировать дефекты со ссылкой на сессию AI и используемый промт.

#### 7.1.2 Чеклист

- [ ] AC покрыты позитивные + негативные + edge-сценарии.
- [ ] Тестовые данные не содержат PII без маскирования.
- [ ] Edge-cases проверены вручную (модель пропускает).
- [ ] Регрессионная нагрузка не превышает прежнюю.
- [ ] Нагрузочные тесты не запускаются на prod-инфре.

#### 7.1.3 Эталонный промт

```
Роль: QA-инженер. Контекст: AC из тикета {ID}, файлы тестов {path}.
Задача: предложи список негативных и edge-сценариев, отсутствующих в текущих AC.
Не предлагай позитивные — они уже описаны.
Формат: таблица — сценарий, риск, тип теста (unit / integration / e2e).
Если данных в AC не хватает — сформулируй вопросы к аналитику.
```

## 12. Откат, обработка ошибок, эскалация

Раздел отвечает на вопрос «что делать, когда что-то пошло не так». Stop-rules — §12.3.

### 12.1 Rollback локальных изменений

После неудачной AI-сессии — `git restore .` (unstaged) или `git restore --staged .` (снять индекс). Полный сброс к `HEAD` — `git reset --hard HEAD` (только если нечего терять; проверить `git status` и `git stash list`). Подробности — git-документация.

### 12.2 Действия при галлюцинации модели

- Не принимать diff с выдуманными API / импортами / файлами.
- Указать модели на конкретный пробел: «функция X не существует в файле Y, проверь».
- Если модель повторяется или продолжает галлюцинировать — закрыть сессию, переформулировать промт с более узким scope, открыть новую.
- Зафиксировать в описании PR, что галлюцинация была — для статистики качества AI-применения.

### 12.3 Когда закрыть сессию и начать заново

**Признаки**:

- агент несколько раз (≥3) подряд предлагает действия, которые пользователь отменяет;
- агент ссылается на API / файлы / библиотеки, которые не найти в репо или документации;
- агент пытается выйти за заявленный scope задачи;
- агент запрашивает доступ к ресурсам, не упомянутым в задаче;
- в diff появляются изменения вне затронутых модулей без объяснения;
- контекст переполнен — модель «забыла» начало диалога;
- промт расширялся итеративно — лучше переписать с нуля, явно зафиксировав scope;
- после откатов в репозитории «грязный» state.

**Действия**: отменить последний шаг → закрыть сессию → начать новую на чистом `git status`. При утечке / опасном действии — §12.3.

## 13. Безопасность

### 13.1 Что НИКОГДА не отдавать модели

**Файлы / пути — закрыть через `permissions.read.deny`:**

- **Секреты и переменные окружения** — `.env*` (вкл. `.env.local`, `.env.production`), `secrets/`, `credentials*`.
- **Облачные / kube / docker конфиги** — `.aws/`, `.kube/`, `.docker/config.json`, `wallet/`.
- **SSH-ключи** — `id_rsa*`, `id_ed25519*`, `known_hosts`, `authorized_keys`, содержимое `~/.ssh/`.
- **TLS-ключи и сертификаты с приватной частью** — `*.pem`, `*.pfx`, `*.p12`, `*.jks`, `*.key`.
- **Package manager creds** — `.npmrc`, `.pypirc`, `.netrc`, `.pgpass`.

Готовый glob-конфиг для OpenCode (`permissions.read.deny`) — [OpenCode Guide](./OpenCode%20Guide.md) §9.1. Особенности конкретного инструмента (например, что `read` в OpenCode не уважает `.gitignore`) — [OpenCode Guide](./OpenCode%20Guide.md) §1, §3.2.


### 13.2 TLS и корпоративный прокси

- Установить корневой сертификат банка в систему ([корпоративная инструкция](https://confluence.moscow.alfaintra.net/pages/viewpage.action?pageId=863154122)) — **единственный корректный путь**.
- **Запрещено** отключать проверку TLS любым способом (переменные окружения, флаги CLI, `strictSSL: false`, `rejectUnauthorized: false`). Это создаёт постоянную поверхность MITM-атак, оседает в shell-профилях / Dockerfile / CI и труднообнаружимо.
- При ошибке сертификата — диагностировать и установить недостающий корневой сертификат, не обходить проверку.

### 13.3 Diff review обязателен

Любая правка проходит через permission-диалог с показом diff. `edit: "ask"` для всех корпоративных и закрытых репозиториев — без исключений.

### 13.4 Безопасность MCP

- MCP наследует permissions агента и может вызывать инструменты с побочными эффектами.
- **MCP с write-доступом в production-системы (БД, очереди, инфраструктура, payment-API) — `deny` по умолчанию.** Запись в production через AI-агента не допускается.

## 17. Модульность репозитория

Если репозиторий разбит на модули (монорепо, мультипакетный проект, несколько сервисов в одной кодобазе), правила и AI-сессии строятся по двум принципам.

**Правила распределяются по уровням:**

- **Корневой** `AGENTS.md` / `CLAUDE.md` — общие правила репо: стиль, ветки, security, deny-rules, провайдеры, MCP, audit-trail.
- **Поддиректорный** `<module>/AGENTS.md` — правила конкретного модуля: стек, конвенции, тесты, точки входа, локальные запреты.
- Поддиректорный файл переопределяет корневой **только в части своего scope**. Security и deny-rules корневого **не ослабляются**.
- Список одобренных MCP и провайдеров — на уровне репо/организации, не модуля. Поддиректорное расширение этого списка запрещено.

**AI-сессия ведётся внутри одного модуля:**

- Scope промта — модуль и его явные зависимости. Перед `Build`-фазой агент читает оба файла правил: корневой и поддиректорный.
- При конфликте правил действуют корневой security/deny-rules + локальные дополнения; локальные правила не могут ослаблять корневые.
- Изменения в чужом модуле в той же сессии — антипаттерн. Кросс-модульные правки — отдельная сессия с явным расширением scope или ADR (см. §3.2).

**Поддиректорный `AGENTS.md` создаётся только если** модуль использует другой стек, имеет собственный набор конвенций или изолирован security/compliance-периметром (PCI/PII/DBO). Иначе достаточно одного корневого файла; дублирование общих правил в поддиректорных файлах — антипаттерн.
