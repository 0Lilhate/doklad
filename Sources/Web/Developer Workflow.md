# Developer Workflow

> ℹ️ **Info — Контекст**
> Полный workflow разработчика для работы с AI coding agents в enterprise-проекте (Java/Spring/Kafka/БД/микросервисы). Общие концепции, security и чеклисты — в [AI Agents](./AI%20Agents.md). Vendor-specific детали (синтаксис subagent'ов, конфиг AlfaGen, MCP) — в [OpenCode Guide](./OpenCode%20Guide.md).

Раздел про то, как разработчику использовать AI-агента в ежедневной работе — не как «чатик для генерации кода», а как помощника, который анализирует, планирует, реализует, проверяет.

---

## 1. Принципы

Базовые принципы работы с AI-агентом (Plan first / маленькие diff'ы / diff-review / контекст важнее промта) — **[AI Agents](./AI%20Agents.md) §3** + **§13.5**. Применительно к разработчику:

- **Агент усиливает, а не делегирует.** Он помогает быстрее изучить код, сформулировать план, написать тривиальные части, найти места для проверки. Архитектурные и бизнес-решения не принимает.
- **Ответственность за код — на разработчике.** Фраза «агент так написал» в post-mortem не работает. Подпись в PR = ответственность.

## 2. Разделение ответственности

| Делает агент | Делает разработчик |
|---|---|
| Читает код, ищет реализации, строит цепочки вызовов | Формулирует задачу и ставит ограничения |
| Предлагает план изменений | Принимает план, редактирует scope |
| Пишет минимальный diff по плану | Читает diff, проверяет логику |
| Генерирует boilerplate (DTO, mapper, конфиги) | Валидирует |
| Пишет unit-тесты под одобренный код | Пишет/правит integration-тесты, проверяет покрытие |
| Запускает локальные команды (build, test, grep) | Решает, что коммитить и куда |
| Описывает diff в человеческом виде для PR | Отвечает на ревью, утверждает мердж |
| Анализирует логи, предлагает гипотезы | Принимает гипотезу, проверяет в проде/стенде |

Агент **не делает**: архитектурные решения, code review за человека, мердж, релиз, общение со стейкхолдерами.

## 3. Базовый цикл (10 шагов)

```
1. Постановка       — бизнес-задача, ограничения, DoD.
2. Сбор контекста   — тикет, требования, смежные модули, контракты.
3. Анализ кода      — read-only режим, найти где менять.
4. Уточнения        — вопросы к аналитику/архитектору, если контракт неясен.
5. Декомпозиция     — план как chain of small commits.
6. План изменений   — explicit scope, список файлов, риски.
7. Реализация       — минимальные diff'ы, по одному файлу за раз.
8. Самопроверка     — agent describes diff, перечисляет риски.
9. Тесты            — unit + integration + manual sanity.
10. PR/MR           — описание, ссылка на тикет, чеклист.
```

## 4. Перед началом задачи (входной артефакт)

Что разработчик должен дать агенту в первом сообщении:

- **Бизнес-задача**: что и зачем делаем, в одном-двух предложениях.
- **Ссылки на источники**: тикет (Jira), требование (Confluence), BPMN, OpenAPI/WSDL, контракт.
- **Целевые модули**: где предположительно нужно менять (`payments-service/api/`, `notifications-worker/`, …).
- **Запреты**: что НЕ менять (`db/migrations/V20240101__*.sql`, `legacy/billing/**`, публичные REST-контракты без ревью архитектора).
- **Ограничения окружения**: версия Spring Boot и Java, особенности корпоративного контура (allow-list зависимостей, обязательные JSON-логи, обязательный `gitleaks` перед коммитом).
- **Ожидаемый формат результата**: «План + diff + тесты», «PlantUML + ADR в `docs/architecture/decisions/`» и т.п.

Чем лучше входной артефакт, тем меньше итераций. Если поставить задачу одной фразой «сделай PROJ-1234» — агент будет угадывать, и почти всегда неверно.

## 5. Анализ существующего кода

Что агент должен сделать (промт — §14):

- Изучить структуру модулей: какие сервисы, где точки входа (`@RestController`, `@KafkaListener`, scheduled jobs, CLI).
- Построить цепочку вызовов от точки входа до места правки.
- Выделить доменную модель (Aggregate, Entity, ValueObject) — отделить core domain от infrastructure.
- Найти интеграции: HTTP-клиенты, Kafka producer/consumer, scheduler, JPA-репозитории, кеши.
- Найти **похожие реализации** — обычно в репо уже решали похожую задачу, и стиль решения должен быть согласован.
- Отличить актуальный код от deprecated/legacy: даты последних коммитов, `@Deprecated`, упоминания в `AGENTS.md`.
- **Не делать выводов без проверки**: если агент пишет «класс X используется только тут», требуем подтвердить grep / `lsp references` со списком найденных мест.

Антипаттерн: «прочитал README → сделал вывод о структуре». README часто устарел — смотреть код.

## 6. Реализация изменения

Базовые правила (план перед кодом, минимальный diff, не трогать публичные контракты в обычной фиче, не уходить в архитектурный рефакторинг) — [AI Agents](./AI%20Agents.md) §3 + §3.1. Dev-specific:

- **Стиль проекта сохраняется.** Naming, форматирование, паттерны исключений, логирование, аннотации валидации — как в существующем коде.
- **Зависимости не добавляются.** Если без новой библиотеки нельзя — отдельный пункт в плане + согласование. В корпоративных контурах есть allow-list для зависимостей.
- **Без «красивого рефакторинга» внутри бизнес-задачи.** Хочешь рефакторить — отдельная задача, отдельный PR с label `refactor`.
- **Каждое изменение связано с требованием.** Если в diff есть строка, которую нельзя обосновать ссылкой на требование — её надо удалить.

## 7. Тестирование

Агент полезен в:

- **Поиск существующих тестов** под изменяемый класс (`grep`, `glob` по `*Test.java` / `*IT.java`).
- **Генерация unit-тестов**: agent видит публичный интерфейс класса и предлагает позитивные / негативные / граничные кейсы. Разработчик принимает или редактирует.
- **Генерация integration-тестов**: `WebMvcTest`, `DataJpaTest`, Testcontainers (Postgres, Kafka). Здесь обязательно дать агенту контекст о существующих helper-классах и тестовом окружении — иначе он напишет «параллельную» инфраструктуру.
- **Edge cases**: попросить вернуть список граничных условий (null, пустая коллекция, отрицательные числа, конкурентный доступ, idempotency, повторная доставка событий Kafka, дубликаты в БД).
- **Тест-кейсы для ручной проверки**: список «что проверить в Postman / на стенде».
- **Анализ логов**: дать агенту лог падения, попросить вернуть гипотезы и точки в коде.
- **Запуск команд сборки и тестов**: `./gradlew test`, `mvn verify`, `./gradlew :module:integrationTest`. Агент видит вывод и интерпретирует.

Антипаттерн: «агент написал тест → коммитим». Тест должен покрывать **реальное** поведение, а не зеркалить (возможно, ошибочную) реализацию. Запускать тест против пред-фиксной версии — он должен падать.

## 8. Code review (self-review перед MR/PR)

Перед отправкой PR использовать агента для:

- **Self-review дифа**: «прочитай мой diff, перечисли что изменилось, что может сломаться, что выглядит подозрительно».
- **Проверка лишних изменений**: автоформатирование, случайные правки в файлах, не относящихся к задаче — частая причина шума в PR.
- **Проверка совместимости с требованиями**: agent сверяет diff с тикетом / Confluence-описанием.
- **Поиск потенциальных багов**: race conditions, NPE, утечки ресурсов, обработка checked exceptions, корректность транзакций (`@Transactional` с self-invocation, propagation).
- **Читаемость**: длинные методы, сложные условия, плохие имена.
- **Подготовка описания PR**: bullet-list изменений, ссылки на тикет, тест-план, миграции / feature-флаги.
- **Подготовка ответов на типичные вопросы ревьюера**.

Self-review агентом **не заменяет** человеческое ревью.

## 9. Багфикс

Отдельный сценарий, отличающийся от feature-задачи:

1. **Ввод**: лог ошибки / стек-трейс / репродуцируемые шаги.
2. **Гипотезы**: попросить агента вернуть 3–5 ранжированных гипотез о причине, без правок кода.
3. **Сужение**: для каждой гипотезы — какие файлы / методы проверить, какие условия должны выполниться.
4. **Первопричина**: найти именно root cause, не симптом. Признак: исправление root cause не требует try/catch вокруг ошибки.
5. **Минимальный фикс**: точечный, с unit-тестом, который воспроизводит баг до фикса.
6. **Проверка**: тест, который раньше падал, теперь зелёный; нет регрессии в смежных тестах.
7. **Обоснование в PR**: явно «Root cause: …, Fix: …, Why this is correct: …».

Антипаттерн: «обернуть в try/catch и залогировать». Это лечение симптома, не причины.

## 10. Работа с legacy

Корпоративный legacy (Spring 4 / Java 8 / монолит, generated stubs, неподдерживаемые библиотеки) — самый рискованный участок.

- **Понять поведение до изменения**: что именно делает класс/метод сегодня. Без этого любой рефакторинг сломает прод.
- **Слепо не рефакторить**. Даже если код «плохо написан», он мог быть таким намеренно (workaround регрессии, требования compliance).
- **Искать неявные зависимости**: рефлексия, AOP, профили Spring, аннотации `@ConditionalOnProperty`, скрытые SPI, JMX-бины, side-effects в статических инициализаторах, `@PostConstruct`-цепочки.
- **Backward compatibility**: публичные методы, JSON-структуры событий Kafka, БД-схема — не ломать.
- **Покрыть тестами изменяемую область до правки**. Если тестов нет — написать характеристические тесты (Michael Feathers, *Working Effectively with Legacy Code*) для текущего поведения. Только после этого менять.
- **Зафиксировать найденное знание** в `AGENTS.md` модуля или в отдельном `docs/<module>/notes.md`. Каждая правка обогащает контекст для следующих.

## 11. Архитектурные задачи

Когда задача больше «фичи» (несколько сервисов, контракты, миграции, выбор технологий) — переключаемся в режим больших задач: **[AI Agents](./AI%20Agents.md) §3.2** (планирующие артефакты, чеклист готовности к большой AI-сессии) и **[Architect Workflow](./Architect%20Workflow.md)** (16-шаговый цикл, ADR, C4, impact analysis, trade-offs, декомпозиция).

Что делает разработчик в архитектурной работе:

- готовит черновик ADR / C4 / impact-анализ как материал для архитектора — **не утверждает решение**;
- после утверждения отвечает за реализацию, не за выбор варианта;
- задаёт архитектору конкретные вопросы вместо того чтобы принимать «общеотраслевой best practice» от агента.

Антипаттерн «vibe architecting» — [AI Agents](./AI%20Agents.md) §14.

## 12. Документация

Агент полезен для:

- **README модуля**: запуск локально, переменные окружения, профили, основные команды.
- **API-документация**: на основе кода контроллеров и DTO — описание эндпоинтов, форматов, кодов ответа.
- **Технические решения**: краткое объяснение «почему так сделано» — для будущих разработчиков.
- **Инструкции запуска**: docker-compose, локальный Kafka, миграции БД.
- **Troubleshooting**: типичные ошибки и решения, дополняемые по мере встреч с ними.
- **Синхронизация документации с кодом**: agent проверяет, не разъехались ли README и реальное состояние модуля.

Документацию писать в репозитории рядом с кодом (не только в Confluence) — она ревьюится в PR и не устаревает «незаметно».

## 13. Роли агента

В рамках задачи можно вызывать агента в разных ролях (через subagents — см. [OpenCode Guide](./OpenCode%20Guide.md) §4.1 — или явно в промте):

| Роль | Когда вызывать |
|---|---|
| Senior backend developer | Реализация, рефакторинг, ревью кода |
| System analyst | Разбор требований, BPMN, OpenAPI |
| Architect | Анализ вариантов, ADR, C4, trade-offs |
| QA engineer | Edge cases, тест-планы, негативные сценарии |
| DevOps engineer | CI/CD пайплайны, Docker, k8s манифесты |
| Security reviewer | Поиск уязвимостей в diff'е, проверка работы с секретами, OWASP-чеклист |
| Code reviewer | Self-review перед PR |
| Technical writer | README, документация, описания PR |

Один и тот же запрос, заданный в разных ролях, даст разный результат.

## 14. Промты — хорошие и плохие

**Плохой промт**:

```
Сделай задачу из тикета PROJ-1234.
```

Что не так: нет контекста, нет ограничений, нет ожидаемого формата. Агент будет угадывать и почти всегда неверно.

**Рекомендуемый базовый промт для разработчика**:

```
Контекст:
- Бизнес-задача: <одно-два предложения>.
- Тикет: <ссылка>. Требования: <ссылка на Confluence/BPMN>.
- Сервис: <module/path>. Стек: Spring Boot 3, Java 21, Postgres, Kafka.
- AGENTS.md проекта прочитан.

Задача:
Сейчас НЕ меняй код. Сделай:
1. Анализ: где в репо реализован сценарий <X>, через какие файлы проходит вызов, какие интеграции затрагиваются.
2. План минимальных изменений с пофайловым diff-описанием.
3. Список рисков и edge cases.
4. Список тестов, которые нужно добавить.
5. Список вопросов, на которые нужны ответы до реализации.

Ограничения:
- Не менять файлы вне: <список>.
- Не трогать публичные контракты (REST DTO, Kafka events).
- Не добавлять новые библиотеки.
- Стиль и паттерны — как в соседних модулях.

Формат ответа: markdown с разделами 1–5.
```

**Промт для реализации**:

```
План одобрен. Реализуй пункт <N> плана.
Меняй только: <список файлов>.
Сохрани стиль соседних классов.
Покажи diff. Тесты не запускай — я запущу сам.
```

**Промт для self-review**:

```
Это мой diff <ссылка/вставка>. Прочитай и верни:
1. Что изменилось — суть в 1 предложении на файл.
2. Лишние изменения, не связанные с задачей.
3. Потенциальные баги (NPE, race, ресурсы, транзакции).
4. Совместимость с требованиями <ссылка>.
5. Черновик описания PR: changes / why / test plan.
```

**Промт для тестирования**:

```
Для класса <FQCN> предложи unit-тесты.
Сначала список кейсов (позитивные / негативные / edge),
затем код тестов в стиле существующих в src/test/java/<module>/.
Не трогай другие тесты.
```

**Промт для багфикса**:

```
Лог ошибки:
<вставить stacktrace>
Контекст: <что делал пользователь, какой эндпоинт, окружение>.

Не меняй код. Верни:
1. 3–5 гипотез о причине, ранжированных по вероятности.
2. Для каждой — какие файлы / методы проверить.
3. Какой тест воспроизведёт ошибку до фикса.
4. Минимальный план фикса для самой вероятной гипотезы.
```

## 15. Безопасность и контроль

> ⚠️ **Warning — Применение к workflow разработчика**
> Полные правила — **[AI Agents](./AI%20Agents.md) §13** (что нельзя отдавать модели, PII, провайдеры, TLS, MCP, audit-trail) и **[OpenCode Guide](./OpenCode%20Guide.md) §3.2** (детализированный набор permissions: `edit: "ask"`, deny-паттерны для секретов, deny для разрушительных bash-команд). Dev-уникального содержания нет — правила общие.
>
> Чеклист перед коммитом — §17 этого документа. Ответственность за код — §1 этого документа + [AI Agents](./AI%20Agents.md) §5.

## 16. Антипаттерны

Общие антипаттерны (без рамок задачи, отключение diff-review, vibe architecting, контаминация контекста, передача PII во внешний контур) — **[AI Agents](./AI%20Agents.md) §14**. Dev-specific:

- «Сделай фичу из тикета» без контекста и ограничений → большой неконтролируемый diff.
- Не дать ссылок на смежные модули → агент пишет «правдоподобный, но чужой» код.
- Принимать большие diff'ы (>300 LOC) одним блоком. Разбивать на 50–100 LOC.
- Смешивать фичу и рефакторинг в одном PR — ревью замусоривается.
- Доверять агенту без тестов. AI-сгенерированный код без unit-теста — это гипотеза, не код.
- Использовать агента как «генератор случайного кода». Если не понимаешь, что он сделал, — не коммить.
- Спешка «закоммитить, пока не сломалось» — главная причина регрессий после AI-правок.

## 17. Минимальный чеклист перед отправкой MR/PR

- [ ] `git diff --staged` прочитан построчно.
- [ ] Каждая строка diff обоснована требованием (не «заодно»); нет выдуманных API / импортов / dead code.
- [ ] Нет лишних изменений: автоформатирование чужих файлов, случайные импорты.
- [ ] Стиль соответствует соседним модулям.
- [ ] Публичные контракты не сломаны (REST DTO, Kafka events, gRPC, public API).
- [ ] Новые зависимости — нет, либо явно согласованы.
- [ ] Unit-тесты покрывают позитивные и негативные кейсы.
- [ ] Integration-тесты покрывают сценарий end-to-end (если применимо).
- [ ] Все тесты зелёные локально.
- [ ] Логирование на уровне INFO/ERROR — без лишнего DEBUG в проде.
- [ ] Описание PR содержит: ссылку на тикет, что изменилось, почему, как тестировать, риски, миграции / feature-flags.
